Strong in-process

src/08_Moments/28_WindowedExpectedReturns.jl · 29_WindowedCovariance.jl · 30_WindowedVariance.jl · 31_WindowedCoskewness.jl · 32_WindowedCokurtosis.jl

WindowedExpectedReturns  →  get_window · window_and_weights · factory · iv-slice
WindowedCovariance  →  get_window · window_and_weights · factory · iv-slice (×2: cov+cor)

WindowedCoskewness  →  get_window · window_and_weights · factory · iv-slice
WindowedCokurtosis  →  get_window · window_and_weights · factory · iv-slice

Same 5-line preamble in every mean/cov/cor/coskewness/cokurtosis body. 704 LOC; the windowing logic lives in ~5 places.

After — one deep windowing module, 5 thin declarations

WindowedMoment<E> — one interface

window = get_window(…)
X, w = window_and_weights(…)
inner = factory(estimator, w)
iv-slice · delegate(inner, X)

mean
cov
cor
cosk
cokurt

Each variant declares only the wrapped estimator type and the delegate. Windowing concentrates in one body.

Problem. Five wrappers share an identical struct shape (estimator, w, window) and an identical compute preamble; only the delegated moment call differs. A fix to the iv-slicing or weight handling must land in 5–7 bodies.

Solution. One deep WindowedMoment{E} parameterised by the wrapped estimator, or a @windowed generator — windowing implemented once, the moment delegate chosen by dispatch.

  • locality: windowing bug fixed once
  • leverage: one impl, every moment estimator
  • interface stays tiny (window spec)
  • ≈600 LOC of preamble absorbed
  • new moments get windowing free
  • test surface: one windowing body

DenoiseCovariance(ce, dn, pdm) = PortfolioOptimisersCovariance(ce, MatrixProcessing(order=(:pdm,:dn), pdm=pdm, dn=dn)) — same call sites, one implementation behind the seam.

Problem. Denoise / Detone / Processed are shallow — each module's interface is nearly its whole implementation, and all three are pre-configured instances of the MatrixProcessing pipeline the fourth already owns.

Solution. Make PortfolioOptimisersCovariance + MatrixProcessing the one seam for covariance post-processing; reduce the three to convenience constructors over it (or retire them).

Deletion test. Delete DenoiseCovariance and the caller writes a one-line MatrixProcessing config — complexity moves to the caller, it does not reappear as new logic. Signature of a shallow module.
  • locality: posdef/denoise/detone ordering in one place
  • 3 modules → 3 adapters over one
  • aligns with ADR 0009 step-symbol design
  • ≈530 LOC of skeleton absorbed

3 · Concentrate the validation matrix

Strong in-process

src/01_Base.jl:1985–2200 — assert_nonempty_nonneg_finite_val · assert_nonempty_gt0_finite_val · assert_nonempty_finite_val

Before — a 3 × 5 grid of near-identical bodies

predicate \ typeDictVecPairArrNumPairNumber
≥ 0•••••
> 0•••••
finite•••••

15 bodies. The only deltas: <= vs <, and whether the sign check is present. Every error-message string is hand-repeated.

After — orthogonal primitives behind the same names

assert_nonempty(val, sym)
assert_finite(val, sym)
assert_sign(val, sym; strict)

extract(::Dict)=values · (::VecPair)=2nd · (::Pair)=2nd · (::ArrNum)=elems

Each predicate × the value-extraction dispatch, composed. Three callers keep their names as thin compositions.

Problem. The interface is three function names; the implementation is fifteen copy-pasted bodies. A change to the message format or a missed sign convention has fifteen places to be wrong — and the value-extraction-by-type logic is re-stated in each.

Solution. Factor the two independent axes — predicate (nonempty / finite / sign) and value extraction (Dict / Pair / array / scalar) — into composable primitives; rebuild the three public asserts as one-line compositions.

  • locality: one message format, one sign rule
  • leverage: ~90 call sites, one impl each axis
  • interface unchanged for callers
  • new value-types: one extract method

4 · Deepen the XatRisk constraint builders

Worth exploring in-process

src/20_Optimisation/20_RiskMeasureConstraints/06–09,19 — XatRisk · Conditional · Entropic · Relativistic · PowerNorm (+ each …Range)

Before — the point measure, then its Range twin re-derived

aux vars + bounds  (shared helper ✓)
VaR: z_var, var_risk, big-M, cone
VaRRange: z_var_l/h, var_risk_l/h, big-M ×2, cone ×2
CVaR / CVaRRange · EVaR / … · RVaR / … · PNVaR / …

Each …Range doubles its point twin's variable + cone setup; the base bounds/expression step is already shared. 5,694 LOC across the directory.

After — one tail-builder, range = two tails

build_tail!(model, side, cone, formula)

point = build_tail!(…)
range = build_tail!(lo) + build_tail!(hi)

Each family supplies its cone + risk formula once; the Range form composes two tail calls instead of copying the setup.

Problem. The base bounds/expression step is factored, but each …Range measure re-states its point twin's auxiliary-variable and cone scaffolding for an upper and a lower tail.

Solution. Extract a single tail-builder taking side · cone · risk-formula; express the point measure as one call and the Range as two.

Caveat. The cone and risk-formula genuinely differ per family (NormOne / SOC / PowerCone; Rockafellar–Uryasev vs entropic vs relativistic). The deepening is in the point↔Range axis, not in flattening distinct cones — verify the tail-builder interface stays small before committing.
  • locality: tail formulation in one place
  • Range follows point automatically
  • cone choice stays explicit per family
  • fewer silent big-M divergences

5 · Unify the risk-measure declaration surface

Worth exploring in-process

src/19_RiskMeasures/01_Base_RiskMeasures.jl + per-measure files — risk_input_kind · supports_precomputed_returns · functor arities

Before — one measure's contract scattered across ≥4 seams

struct + @pprop fields
file A
risk_input_kind(r)
one line, elsewhere
supports_precomputed
one line, elsewhere
functors r(x)/r(w,X)
arity must match kind

The three trait declarations and the functor arity must agree, but nothing co-locates or cross-checks them — a mismatch surfaces late, as a wrong route in expected_risk or an optimiser crash.

After — one declaration, emitting the trio

@risk_measure MyMeasure kind=net_returns precomputed=true

⇒ risk_input_kind · supports_precomputed_returns · functor stubs, in sync

Same machinery family as @propagatable / @forward_properties: one declared surface, the introspection generated and kept consistent.

Problem. A risk measure's "what do I consume" contract (ADR 0006 Risk Input Kind + ADR 0007 precomputed contract + functor arity) is spread across three trait methods plus the functors, with no single place to read or verify it.

Solution. A @risk_measure declaration adjacent to the struct that emits the trait methods and functor stubs from one statement — the same "declared surface" discipline ADRs 0012/0013 applied to binding and forwarding.

  • locality: one contract per measure
  • kind ↔ arity checked at expansion
  • new measures can't omit a trait
  • fits the existing macro family

6 · Gerber / Smyth-Broby marker types

Speculative in-process

src/08_Moments/05_GerberCovariance.jl · 06_SmythBrobyCovariance.jl

Before — combinatorial marker structs

Gerber0/1/2 SmythBroby0/1/2 …Gerber0/1/2 …Count0/1/2

~12 fieldless markers; the algorithmic delta is a few lines of which boolean matrices to combine. The types name a cross-product, not state.

After — axes as data (à la ADR 0009)

GerberAlgorithm(base=:sb0, cross=:gerber, count=true)

ADR 0009 already turned ordering markers into step symbols. The same move could fold the cross-product into a small data-carrying algorithm.

Problem. The variant explosion is a cross-product expressed as a type lattice; adding an axis multiplies the markers. Solution. Carry the axes as fields on one algorithm, dispatching internally.

Why only speculative. Unlike the matrix-processing order (a thin meta-choice), these variants are genuinely distinct co-movement algorithms, and the type lattice gives precise dispatch and docstring anchors. The win is real but the current design is defensible — confirm the variants share enough body to be worth merging before touching it.

Top recommendation

1 · Collapse the Windowed moment family

Highest leverage-to-risk. Five modules with a byte-identical struct shape and an identical windowing preamble — pure copy-paste, no divergent edge cases to preserve, isolated under 08_Moments so the blast radius is contained. Deepening it concentrates the windowing implementation in one place (locality) and makes every present and future moment estimator windowable through one tiny interface (leverage). It is also the cleanest precedent for the next two — covariance pipeline and validation matrix — which share the same "one implementation, many shallow copies" shape.

Vocabulary per the review glossary: module · interface · implementation · depth · seam · adapter · leverage · locality. Domain terms per CONTEXT.md. ADRs 0001–0013 honoured, not re-litigated.
