PortfolioOptimisers.jl — deepening opportunities

2026-06-10 · branch dev-improve-arch · ~89k LOC / 172 files · informed by CONTEXT.md and ADRs 0001–0005

module seam leakage / duplication deep module

Five candidates, ranked. Each turns a cluster of shallow modules — interface nearly as wide as the implementation — into one deep module: more behaviour behind a smaller interface, so changes, bugs, and tests concentrate in one place. The verified centrepiece is the duplicated JuMP model-assembly pipeline. None of these contradict ADRs 0001–0005; two of them extend the philosophy of ADR 0001.

Optimisers inject 3 hooks. The 14-step middle lives once.

Problem

Every JuMP optimiser re-writes the same constraint-ordering sequence inline; the assembly is not a module, so it has no interface to test or change in one place.

Solution

Lift the shared middle into one assemble_jump_model! module. Each optimiser supplies three thin hooks: how weights are shaped (set_w! vs set_risk_budgeting_constraints! vs set_factor_risk_contribution_constraints!), the objective, and the solve/finalise tail.

Evidence

11_MeanRisk.jl:631–656 and 14_RiskBudgeting.jl:688–713 — same 18 calls, same order. FRC (12:299–341) & NOC (13:1018–1104) repeat it again with their own head/tail.

Wins

  • locality: constraint order lives once
  • leverage: one interface, N optimisers
  • new constraint type → edit one module
  • interface testable without a solver: assert the built model, don't solve it
  • meta-optimisers reuse the same assembler
Complements ADR 0004/0005. The typed model-state interface (get_w, get_k, prefix-namespaced risk state) is the accessor behind each step; this candidate is the orchestration on top. The seam this opens is exactly where ADR 0004 said builders "just know" the population order — assembly makes that order a single named interface.

2 · Let each risk measure classify itself — retire the expected_risk union registry

A new measure must be hand-added to the right Union far away. Miss it → silent fallback to r(w, X), wrong answer, no error.

After — classification stated at the definition site

CVaR

net_returns

EVaR

solver

Variance

weights

expected_risk dispatches on one trait:
risk_input_kind(::RiskMeasure)

Each measure declares its own input shape. Adding a measure touches only its own file. A missing declaration is a method error, not a wrong number.

Problem

Whether a measure needs net returns, raw weights, or a solver is a fact about that measure, but it's recorded in four ever-growing Union aliases in a distant file. The interface to "compute risk" leaks the full census of measures.

Solution

Replace the unions with a single trait method each measure answers (risk_input_kind). expected_risk dispatches on the trait. The catch-all fallback becomes an explicit, loud error.

Evidence

26_ExpectedRisk.jl:25–124 — ERkNetRet lists 36 types by hand; ERkwXFees/ERkw/SlvRM add ~23 more. Line ~231 silently routes unregistered measures to expected_risk(r, w, pr.X).

Wins

  • locality: a measure's I/O shape lives with the measure
  • silent-fallback bug becomes a method error
  • no distant edit when adding a measure
  • interface stops enumerating every measure
Extends ADR 0001/0002. ADR 0001 chose explicit @prop field tagging over central type-dispatch because "the domain decision cannot be recovered from types alone — it must be stated at the definition site." The union registry is the same decision made in the wrong place. This candidate moves it to the definition site, exactly as @prop did for factory fields.

3 · One windowing adapter, not five

08_Moments/28_WindowedExpectedReturns · 29_WindowedCovariance · 30_WindowedVariance · 31_WindowedCoskewness · 32_WindowedCokurtosis

Four identical lines, written five times (and twice each, for cov & cor). Adding a sixth moment means a sixth copy.

After — windowing wraps any estimator once

deep adapter

Windowed(inner)

get_window · slice · factory · delegate

any cov
any μ
any tensor

Slice-and-delegate is the deep part; it sits once. The moment kind is a parameter, not a new type.

Problem

Windowing — pick observations, re-weight, delegate — is one behaviour, but it's re-implemented as five near-identical decorator types, each with duplicated cov/cor bodies.

Solution

One Windowed adapter parameterised by its inner estimator. The slice-and-delegate logic lives once; moment_view/factory forward through generically.

Evidence

29_WindowedCovariance.jl:152–197 — cov and cor are the same body twice. 30/31/32 repeat it for Variance, Coskewness, Cokurtosis; 28 for ExpectedReturns.

Wins

  • interface shrinks five types to one
  • locality: windowing edited in one body
  • leverage: works for any future moment
  • one test exercises the seam for all
Deletion test, honestly. Deleting a Windowed type pushes the slice-and-delegate dance onto callers — so each one is earning its keep; it is a decorator, not a pass-through. The friction is not shallowness but five-fold duplication of one deep behaviour. The collapse is a leverage/locality win, not a deletion.

Each wrapper re-implements the same cov → ensure-mutable → posdef! → process seam. 07_MatrixProcessing already composes denoise+detone+posdef in a configurable order — the wrappers duplicate a slice of it.

Problem

Three covariance wrappers each hand-roll the same "estimate, then post-process the matrix" orchestration that 07_MatrixProcessing already owns — the post-processing seam leaks into 08_Moments three times.

Solution

One ProcessedCovariance(inner, matrix_processing) adapter that estimates then defers to the existing 07_MatrixProcessing order. Denoise-only / detone-only become configurations, not separate types.

Evidence

12_DenoiseCovariance.jl:170–182 — cov does dims-check, transpose, Matrix(), posdef!, denoise! by hand; 13/14 repeat with detone/process. 07_MatrixProcessing.jl already sequences all three.

Wins

  • post-processing seam crossed once
  • locality: order logic lives in 07 only
  • denoise/detone/custom freely composable
  • two adapters justify the seam (cov + raw matrix)
Speculative in-process

5 · Lift the per-measure settings & bounds plumbing behind one interface

20_Optimisation/19_RiskMeasureConstraints/01_BaseRiskConstraints.jl · 02_VarianceConstraints.jl (+ ~20 measure files)

Before — interface as tall as implementation

one measure

interface (settings, bounds, weight-select) impl

Each measure repeats nothing_scalar_array_selector + get_observation_weights and ends with set_risk_bounds_and_expression!. The variance-named variance_risk_bounds_val is called generically by Kurtosis, Skewness, Tracking.

After — thin interface, behaviour absorbed

one measure

Each measure declares only its risk expression. Weight-selection and bound-transform (renamed off "variance") move into the shared set_risk_bounds_and_expression! entry, called once per measure with no setup.

Problem

The settings/scale/bounds/weight-selection plumbing is copied into the head and tail of ~20 measure constraint builders; a variance-named bounds transform is reused generically, a leaky name.

Why speculative

Some of this overlaps candidate 1's spirit and may be better tackled after it. The bound transform genuinely varies per measure (sqrt for SOC, identity for quantile), so the seam must stay configurable — risk of over-abstracting.

Evidence

71 calls to set_risk_bounds_and_expression! across 20 files; variance_risk_bounds_val (02_VarianceConstraints.jl:435–454) called from Kurtosis/Skewness/Tracking; nothing_scalar_array_selector+get_observation_weights repeated ~15×.

Wins

  • interface shrinks; impl absorbs plumbing
  • bound transform gets a measure-neutral name
  • weight-selection defined once

Top recommendation

Start with candidate 1 — collapse the JuMP model-assembly pipeline.

It is the only candidate verified line-for-line as duplicated rather than merely repetitive: MeanRisk and RiskBudgeting build the identical 18-step model, and FRC/NOC repeat it. It is pure in-process work, so the new module is tested directly by inspecting the assembled model — no solver in the loop. It sits cleanly on top of the ADR 0004/0005 typed model-state interface, and it unlocks candidate 5 and the meta-optimiser cleanup. Highest locality and leverage per unit of risk.

Next-best independent win: candidate 2 — it is small, self-contained, removes a silent-fallback bug class, and continues the ADR 0001 "decide at the definition site" philosophy.
