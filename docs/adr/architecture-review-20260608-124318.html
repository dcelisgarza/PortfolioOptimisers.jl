1 · Give the JuMP model a typed state interface

Strong in-process
src/20_Optimisation/08_Base_JuMPOptimisation.jl
src/20_Optimisation/09_JuMPConstraints/*.jl  (10 files)
src/20_Optimisation/19_RiskMeasureConstraints/*.jl  (20 files · 53 set_risk_constraints! methods)

Before — untyped blackboard

Every builder reaches raw into the shared dict. 534 references · 112 raw model[:x]= writes.

After — one typed seam

Symbol conventions + "is it populated yet" invariant live behind one interface.

Problem. The JuMP model is the seam between builders, but it has no interface — 120 raw symbol keys, no accessors, and existence + population-order are invariants every caller must just know.

Solution. A thin module owning the model-state vocabulary: named, checked register/lookup for the returns matrix, weights, scale, net returns, drawdowns.

  • Locality: convention in one place
  • Interface shrinks: 120 keys → named accessors
  • Ordering invariant enforced, not assumed
  • Absent/typo'd-key bugs concentrate
  • Deletion test: passes — delete it, complexity reappears across 534 sites
  • The seam already exists physically; it just lacks an interface

After — macro absorbs the amber

@riskmeasure absorbs ctor + validation

factory — stays bespoke
risk_measure_view — stays bespoke

@propagatable already proves the project's appetite for this.

Caveat — why Speculative. factory, risk_measure_view and the functor are genuine multiple dispatch, not boilerplate: the deletion test says complexity reappears if removed. Only the forwarding ctor + standard validation is mechanical. The gain is real but modest, and a macro trades line-count for a layer of indirection.

Problem. Across 24 measures the validating inner ctor + forwarding keyword ctor repeat; the genuine per-type content (functor/factory/view) is buried under scaffold.

Solution. A @riskmeasure macro generating the forwarding ctor + standard field validation, leaving functor/factory/view bespoke.

Top recommendation

Start with #1 — the typed JuMP model state.

It is the leakiest seam in the codebase and the only candidate whose friction is independent of Julia idiom: 120 raw symbol keys, 534 references, zero accessors, and ordering invariants every one of the 53 set_risk_constraints! methods must silently assume. It also unblocks every future risk measure — today that means knowing which of 120 keys already exist. Candidates 2–4 are about naming and scaffolding; #1 is about a seam that is actively leaking.
