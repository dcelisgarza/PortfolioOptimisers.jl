01 · Deepen moment resolution behind one interface

Strong in-process
src/19_RiskMeasures/{02_Variance, 04_Kurtosis, 06_XatRisk, 08–10, 11, 19, 20, 24}.jl
src/20_Optimisation/19_RiskMeasureConstraints/02_VarianceConstraints.jl:341,365,602,727
src/20_Optimisation/19_RiskMeasureConstraints/20_VarianceSkewKurtosisConstraints.jl:37–39

Before — fallback inlined at every call site

# 02_VarianceConstraints.jl:341
sigma = isnothing(r.sigma) ? pr.sigma : r.sigma
# …:365, :602, :727 — same line, 4×

# 20_VarianceSkewKurtosisConstraints.jl:37
sigma = ifelse(isnothing(r.vr.sigma), pr.sigma, r.vr.sigma)
sk   = ifelse(isnothing(r.sk.sk), pr.sk, r.sk.sk)
kt   = ifelse(isnothing(r.kt.kt), pr.kt, r.kt.kt)

~12 measure files + constraint files each re-implement "use the measure's override field, else fall back to the Prior." The constraint builder reaches across the seam into sub-measure fields (r.vr.sigma) and the Prior.

After — one deep resolver

# one interface, dispatched per measure
sigma = resolve_sigma(r, pr)
sk   = resolve_sk(r, pr)
kt   = resolve_kt(r, pr)

# composite delegates to its parts:
resolve_sigma(r::VarSkewKurt, pr) =
  resolve_sigma(r.vr, pr)

The override-or-Prior rule lives once. Composites compose resolvers instead of knowing their parts' field names. A renamed field breaks one method, not twelve.

Problem

The override-or-Prior moment rule is inlined at ~20 sites, and composite constraint builders read sub-measure fields directly — leakage across the measure seam.

Solution

One resolve_*(measure, prior) per moment; composites compose the resolvers of their parts.

  • locality: fallback rule lives in one module
  • leverage: one interface, ~20 call sites
  • field rename breaks one method, not twelve
  • interface shrinks; the resolver absorbs the ifelse
  • tests hit the resolver, not each builder
  • deletion test: concentrates

The shared run-inner / rebuild / finalise plumbing becomes a deep module each meta-optimiser configures with its own indexing.

The cross-validation aggregation harness is owned by one meta-optimiser's file but consumed by the other two — poor locality and a hidden dependency on NestedClustered's internals.

⚠ Echoes ADR-0008 (model-assembly pipeline) — same move, different layer. The three inner loops differ only in indexing (clusters / stacks / subsets); confirm that variance fits one strategy parameter before committing.
  • locality: CV aggregation in one module
  • Stacking stops depending on NCO's file
  • new meta-optimiser configures, doesn't copy
  • deletion test: concentrates

05 · Make the correlation matrix the Distance interface

Worth exploring in-process
src/09_Distance/02_Distance.jl:122–267 (ce, X) paths
src/09_Distance/02_Distance.jl:319–405 (rho) paths · :439+ cor_and_dist

Before — each algorithm coded twice

distance(de, ce, X) · computes rho, forwards
distance(de, rho) · the real formula
cor_and_dist(de, ce, X) · computes rho, forwards
…× ~6 algorithms × {Nothing, Integer}

The primary path forces a CovarianceEstimator; the pure correlation→distance formula is the secondary overload. Testing a distance formula drags in moment estimation.

After — one formula, one thin adapter

distance(de, rho) · the formula, the test surface
distance(de, ce, X) = distance(de, cor(ce, X))

Each algorithm's formula exists once. The estimator path is one adapter line. Distance is testable from a bare correlation matrix.

Problem

Distance algorithms are implemented per input shape; the (ce, X) methods are pass-throughs that compute a correlation then delegate, and they make the formula hard to test in isolation.

Solution

Treat distance(de, rho) as the interface; reduce the estimator paths to a single shared adapter that computes rho once.

  • interface: the correlation matrix, one shape
  • test surface: formula without moments
  • collapses ~6 pass-through methods
  • deletion test: concentrates

One module per measure; the JuMP builder is an internal seam, not a parallel file. Locality: the whole measure reads top to bottom.

Problem

Each measure is split across two parallel files; the definition and its formulation evolve together but live apart, coupled by shared key names.

Solution

Co-locate definition + functor + JuMP builder per measure, keeping the builder as an internal seam.

⚠ The split may be load-bearing: it keeps the JuMP dependency on one side and lets the functor side stay solver-free (relevant to ADR-0006 / ADR-0007). Treat as a question to answer, not a refactor to schedule — measure whether anything actually imports the functor side without JuMP before moving.
  • locality: one measure, one module
  • shared key names become an interface
  • risk: may collapse a real solver-free seam
  • deletion test: unclear — verify first

Top recommendation

Start with 01 · Moment resolution

It is the widest, lowest-risk deepening: the override-or-Prior rule is duplicated across ~20 sites in ~12 files, and composite constraint builders reach across the measure seam into sub-measure fields. One resolve_*(measure, prior) interface concentrates that rule, lets composites compose their parts' resolvers, and turns a renamed field from twelve breakages into one. Pure leverage and locality, no JuMP or solver risk, and it sets up the cleaner seam that candidate 06 would later test.

Vocabulary: module · interface · implementation · depth · seam · adapter · leverage · locality. Decided in ADRs 0001–0008 and deliberately excluded.
