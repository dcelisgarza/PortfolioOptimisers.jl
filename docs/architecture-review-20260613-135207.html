1 · Auto-generate moment_view like factory

Strong in-process
  • src/08_Moments/ — 29 hand-written moment_view methods (every estimator)
  • src/02_Tools.jl:706 — @propagatable macro + @prop field tags (ADR 0002)

Before — 29 near-identical methods

moment_view(ce::Covariance, i) =
  Covariance(; me=moment_view(ce.me,i), ce=moment_view(ce.ce,i), alg=ce.alg)
moment_view(ve::SimpleVariance, i) =
  SimpleVariance(; me=moment_view(ve.me,i), w=ve.w, corrected=ve.corrected)
moment_view(ce::DenoiseCovariance, i) =
  DenoiseCovariance(; ce=moment_view(ce.ce,i), dn=ce.dn, pdm=ce.pdm)

⋮

× 29, one per estimator

Recurse into the data-bearing child fields, pass the rest through. A field forgotten is a silent wrong slice.

After — emitted from the same @prop tags

@propagatable @concrete struct Covariance
  @prop me
  @prop ce
  alg
end
↓ macro emits both
factory(x, w::ObsWeights)
moment_view(x, i)

One @prop tag set drives weight propagation and windowing. Interface unchanged; 29 bodies absorbed.

Problem

moment_view is the same pure-propagation shape factory had before ADR 0001/0002 — recurse into the same @prop child fields — but got no macro; 29 methods are hand-maintained.

Solution

Extend @propagatable to also emit moment_view(x, i) from the existing @prop tags.

  • locality: slice logic in one macro
  • leverage: one tag set, two functions
  • interface unchanged; 29 bodies absorbed
  • field-omission bug becomes impossible
Builds on ADR 0002, doesn't contradict it. Same @prop seam, second consumer. Two consumers (factory, moment_view) now justify the tagging machinery that one consumer already paid for.

# MeanRiskResult, RiskBudgetingResult, NaiveOptimisationResult … identical:
function Base.getproperty(r, sym::Symbol)
    return if sym == :w
        !isa(r.sol, AbstractVector) ? getfield(r.sol, :w) : getfield.(r.sol, :w)
    elseif sym in propertynames(r);    getfield(r, sym)
    elseif sym in propertynames(r.pa); getproperty(r.pa, sym)   ← FRC inserts r.rr before r.pa
    else                               getfield(r, sym)
    end
end

Problem

The :w-virtual + forward-to-.pa interface is identical across 10 result types, but the body is copy-pasted; the only real variation is the forward chain order.

Solution

A deep result module: one getproperty + factory(res, fb) over a declared forward-chain tuple. Per-optimiser types add only their unique fields.

  • locality: :w extraction fixed once
  • interface shrinks; core absorbs the copies
  • the forwarding becomes the test surface
  • leverage: one core, 10 result types

The Vector-to-Scalar Reducers are a named domain family in CONTEXT.md §1, marooned beside generic utilities and macro AST code.

Problem

Three concerns with nothing in common share one 1620-line file: low-level utilities, the @propagatable macro, and the VectorToScalarMeasure reducer family. No locality — finding the reducers means scanning past macro AST helpers.

Solution

Lift the reducer family into its own module (a named module for a named domain concept). Optionally isolate the @propagatable machinery too, leaving 02_Tools as pure utilities.

  • locality: domain family in one module
  • AI-navigability: file name = concept
  • adding a reducer touches one small module
  • macro machinery stops diluting utilities
This is relocation for locality, not a depth change — the reducers are already deep. Lower payoff than 1 & 2; do it when next editing the file.

Speculative ports & adapters

Each measure's field layout and formulation algorithms are split across a parallel file pair. Adding a measure means coordinated edits in two trees.

Problem

20 parallel file pairs. A new measure, or a new formulation algorithm on an existing one, requires synchronised edits in both trees — and the constraint file must know the measure's internal field layout.

Solution (tentative)

Co-locate each measure's definition and its JuMP formulation, keeping the JuMP dependency behind the same seam — definition usable without a solver, formulation an adapter loaded with JuMP.

Tension with ADRs 0006 & 0007. The split deliberately keeps JuMP out of measure definitions — measures evaluate through functors (risk_input_kind, supports_precomputed_returns) with no solver. The seam is real: JuMP-formulation is a genuine adapter. So this is co-location for navigability, not a seam to remove — and only worth it if the cross-tree sync is actually causing bugs. Surfaced for discussion, not recommended.

Top recommendation

Start with 1 — auto-generate moment_view.

Highest leverage per unit of risk. The @prop seam already exists and is trusted (ADRs 0001/0002); adding moment_view as a second consumer absorbs 29 hand-written bodies, removes a whole class of silent field-omission bugs, and changes no caller's interface. It deepens an existing module rather than introducing a new one. Candidate 2 is the natural follow-up — same shape (10 copies of one behaviour), slightly larger blast radius.
