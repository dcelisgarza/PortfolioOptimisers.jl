A · MeanRisk() advertises a default it cannot build

A
Files

src/20_Optimisation/11_MeanRisk.jl:235
src/20_Optimisation/10_JuMPOptimiser.jl:738

Problem → Solution

The signature promises opt = JuMPOptimiser(), but that call throws.
Give slv a default, or drop the unreachable default on opt so the requirement is honest.

Before — the trap

function MeanRisk(; opt::JuMPOptimiser = JuMPOptimiser(),
                  r::RM_VecRM = Variance(),
                  obj::ObjectiveFunction = MinimumRisk(), …)

function JuMPOptimiser(; pe::PrE_Pr = EmpiricalPrior(),
                       slv::Slv_VecSlv,   # ← required, NO default
                       wb = WeightBounds(), bgt = 1.0, …)
julia> MeanRisk()
ERROR: UndefKeywordError: keyword argument `slv` not assigned

After — honest signature

function JuMPOptimiser(; pe = EmpiricalPrior(),
                       slv::Slv_VecSlv = Solver(),
                       wb = WeightBounds(), bgt = 1.0, …)

…or, if a solver genuinely can't be defaulted, make the dependency visible at the top level instead of hiding it three constructors down:

MeanRisk(; slv, opt = JuMPOptimiser(; slv), …)

Either way MeanRisk() with the documented defaults must run, or the default must not be documented.

Default test: every JuMP caller passes a solver — and the same broken pattern recurs at Solver(; solver = nothing) (Card F). The friction is hidden because the failing default lives three constructors below the one the caller typed.
✓ default that constructs ✓ honest signature ✓ failure at call site, not 3 levels down ✓ matches HRP, which does construct

B · Half the argument checks name a variable the caller never typed

Problem → Solution

A bare check echoes the failed Julia expression — internal names like r, wi, val — never the caller's argument or what to pass.
Adopt the project's own good pattern across the surface: a typed error with a caller-facing sentence.

Before — what the caller sees

@argcheck(!isempty(r))            # HRP:162
@argcheck(length(wi) == length(w)) # JuMP:612
ERROR: ArgumentError: !isempty(r) must hold.
Got !isempty(r) => false

# what is `r`? the caller passed a risk measure
# to `MeanRisk`, never an `r`. And `wi` vs `w`?

After — the pattern already in the repo

@argcheck(!isempty(r),
    IsEmptyError("risk measure vector `r` is empty; \
pass at least one risk measure, e.g. r = Variance()"))

@argcheck(length(wi) == length(w),
    DimensionMismatch("initial weights `wi` ($(length(wi))) \
must match asset count ($(length(w)))"))

This is not new machinery — it's the existing convention at 02_Tools.jl:49: IsEmptyError("$names_sym cannot be empty."). The friction is that it holds only ~half the time.

Convention coverage — @argcheck sites with a caller-facing message

404 messaged
401 bare

A caller can't predict which kind of error they'll get — the convention test fails at the worst moment, on a mistake.

C · The …RiskMeasure suffix is true 30% of the time

C
Files

src/19_RiskMeasures/*.jl

Problem → Solution

Some risk measures carry a RiskMeasure suffix; most don't. A caller can't predict the name and can't trust autocomplete.
Pick one rule and add deprecated aliases for the minority that change.

Bare — no suffix (the majority)

Variance Kurtosis NegativeSkewness AverageDrawdown MaximumDrawdown ConditionalValueatRisk WorstRealisation Range

Suffixed — …RiskMeasure (the minority)

TurnoverRiskMeasure TrackingRiskMeasure RiskTrackingRiskMeasure EqualRiskMeasure RiskRatioRiskMeasure

Say them aloud: Variance and TurnoverRiskMeasure are the same kind of thing. The signal disagrees with itself.

ADR note: ADR-0006 classifies a measure's input shape with a trait, not subtypes — it does not legislate the surface name. A naming convention here is open, not already-decided. Worth recording the chosen rule as an ADR so this review doesn't re-suggest it.
✓ predictable from one rule ✓ autocomplete-friendly ✓ aliases keep blast radius low

D · 352 flat exports bury the 2 entrypoints among the macros

D
Files

src/01_Base.jl:2869 · src/02_Tools.jl:2277

Problem → Solution

A newcomer typing PortfolioOptimisers. <TAB> meets 352 names — optimise sits beside @fprop, @vprop, factory, port_opt_view.
Separate the caller surface from the extension surface: export the entrypoints, mark the machinery public.

Before — one flat tier

export @propagatable, @fprop, @vprop, @pprop,
       @cprop, @wprop, @forward_properties,
       factory, optimise, prior, … # 352 names

The signal test: from the export list alone, which name does a caller call first? Nothing distinguishes the 2 entrypoints from the ~7 extension macros from the ~340 building blocks.

After — two tiers (Julia 1.11 public)

# caller surface — exported
export optimise, prior, factory,
       prices_to_returns, MeanRisk, …

# extension surface — public, not exported
public @fprop, @vprop, @pprop, @cprop,
       @forward_properties, port_opt_view

Extenders still reach the macros by qualified name; everyday callers get a curated namespace. No capability is removed — only re-tiered.

Pipeline a caller actually walks (the export list hides this order)

E · Swapping the covariance estimator means threading five constructors

E

Problem → Solution

"Use a Gerber covariance" forces a caller to know and re-state four wrapper types just to reach the ce slot.
A keyword-forwarding convenience constructor (or a factory-style override) for the common one-field swap.

Composition depth to the covariance slot — structure confirmed against the graphify graph

flowchart TD
  MR["MeanRisk"] -->|opt| JO["JuMPOptimiser"]
  JO -->|pe| EP["EmpiricalPrior"]
  EP -->|ce| POC["PortfolioOptimisersCovariance"]
  POC -->|ce| COV["Covariance"]
  COV -->|ce| GC["GeneralCovariance <- the slot you wanted"]
  MR -.->|r  ·  depth 1| RM["risk measure  ✓ shallow"]
  classDef deep fill:#7c2d12,stroke:#fb923c,color:#fff;
  classDef ok fill:#064e3b,stroke:#4ade80,color:#fff;
  class POC,COV,GC deep;
  class RM ok;

Before — five nested constructors

MeanRisk(; opt = JuMPOptimiser(; slv,
  pe = EmpiricalPrior(;
    ce = PortfolioOptimisersCovariance(;
      ce = Covariance(;
        ce = GeneralCovariance(...))))))

After — name the common swap

# risk measure is already this easy — depth 1:
MeanRisk(; r = ConditionalValueatRisk())

# give the covariance swap the same altitude:
MeanRisk(; slv, covariance = Gerber2())

Note the contrast: the risk measure r is a top-level field, so it's a one-liner. The covariance estimator is the same conceptual "one knob," but lives four levels deeper. The asymmetry is the friction.

✓ common swap is one line ✓ matches risk-measure altitude ✓ full tree still available ⚠ define "common" carefully

Problem → Solution

The one field every caller must set — the backend — defaults to nothing, deferring the failure to solve time.
Make solver the leading positional/required argument; keep name et al. defaulted.

Before

function Solver(; name::Sym_Str = "",

A caller who writes Solver(name="clarabel") gets a struct that looks configured and fails only later, far from the typo.

function Solver(solver; name::Sym_Str = "",
                settings = nothing,

                add_bridges::Bool = true)

G · No American-spelling alias for optimise

G

Problem → Solution

Callers arriving from scikit-learn / cvxpy reach for optimize and Optimizer; neither exists.
A thin const optimize = optimise alias — additive, zero blast radius.

Before

julia> optimize(mr, rd)
ERROR: UndefVarError: `optimize` not defined

After

const optimize = optimise
export optimize

British spelling stays canonical (it's the documented convention); the alias just stops a foreseeable dead-end.

Judgement call: dual spellings can themselves be friction (two names for one thing). Reasonable to reject — and if so, worth an ADR so it isn't re-proposed.

Top recommendation

Start with A + B — the two that bite on the first thirty minutes

A newcomer's first session is: build an optimiser, run it, hit a wall. Both walls are here. Card A means the very first MeanRisk() they type — with the documented defaults — throws UndefKeywordError: slv. Card B means their next mistake is answered by an error that names a variable they never wrote. Together they shape the impression that the library is harder than it is.

Both are cheap and reuse patterns already in the repo — a constructible default (HRP already does this) and the IsEmptyError("…") convention from 02_Tools.jl. Neither is a breaking change for correct existing code: A only makes a currently-throwing call succeed; B only enriches error text. C and D are the higher-value but higher-discussion follow-ups (naming convention + export tiering) — best routed through an ADR each.
