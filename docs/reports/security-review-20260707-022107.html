Security review — PortfolioOptimisers.jl

2026-07-07 · branch dev

Fourth pass over the library's trust boundaries, tracing untrusted input — returns/price data, constraint and view strings, asset-set dicts, load-time preferences — to its sinks. The equation-parser surface hardened by ADRs 0025–0027 held up under re-inspection, and two items left open by the 2026-07-06 review are now closed in code. # ReturnsResult inner constructor:
@argcheck(all(isfinite, X),
    IsNonFiniteError("X contains " *
      "non-finite values; impute or " *
      "drop them (see prices_to_returns)"))
# same for F, B — mirrors the iv check
# one line each, every estimator covered

# prices_to_returns transform widened:
x -> ifelse(isa(x, Number) &&
            !isfinite(x), missing, x)
# zero-price Inf now handled like NaN

Problem. Returns data is the library's primary untrusted input, and the path is reachable from both entry points with default settings: prices_to_returns converts NaN to missing but lets Inf through (a zero price yields an infinite simple return), and direct ReturnsResult construction validates finiteness of iv/ivpa but not of X/F/B — so one bad tick reaches the comparison sinks in the Gerber/Smyth-Broby/rank estimators, where NaN comparisons evaluate to false and the observation is silently counted as "no co-movement", producing a finite, plausible-looking correlation and wrong weights rather than a loud NaN.

Direction. Enforce all(isfinite, ·) on X/F/B in the ReturnsResult inner constructor (the one choke point every prior and optimiser already passes through, mirroring the existing iv check) and widen the prices_to_returns NaN transform to all non-finite values so imputation/row-dropping handles them upstream.

  • fails closed on non-finite data
  • closes a silent-wrong-result path
  • one boundary covers every estimator
  • blast radius: ~4 lines, 1 file
Carried over: second half of the open item from the 2026-07-06 review (the allunique half is closed — see baseline). Legitimate missing-data workflows already pass through impute_method/row-dropping before this boundary, so the check does not collide with them.

2 · Empty equation side fails open to zero behind a @warn

Strong Medium input-validation
src/12_ConstraintGeneration/02_LinearConstraintGeneration.jl:902–905 (call sites ≈1084–1090)
Before — truncated config row becomes a real constraint
"AAPL >= "        # cell truncated in the spreadsheet export
→ Meta.parse("") == nothing
→ @warn "rhs of equation is empy, assuming zero"   # log line only
→ constraint AAPL >= 0 enters the optimisation
After — fails closed
"AAPL >= "
→ Meta.ParseError: rhs of equation is empty in "AAPL >= "

Problem. rethrow_parse_error(::Nothing) turns an empty side of an untrusted constraint/view string into an assumed 0 with only a warning — reachable from every string that funnels through parse_equation — so a malformed row from a config file or spreadsheet silently becomes a live constraint (or view) the author never wrote instead of an error they would see.

Direction. Throw a typed Meta.ParseError from the ::Nothing method (two-line change, and fixes the "empy" typo); callers who genuinely want "empty means zero" can write the zero.

  • fails closed on malformed equation
  • no ADR 0026 collision
  • blast radius: 2 lines, 1 site
  • closes review item R4 (2026-07-02)
ADR 0026 does not protect this path: its lenient default exists because meta-optimisers legitimately run constraints against asset subsets where a name may be absent. An empty equation side is never subset-legitimate — failing closed here cannot break those loops.

The library already owns the precedent: CombinatorialCrossValidation takes an overridable max_comb = 100_000 and fails closed past it (03_Combinatorial.jl:66–72).

Problem. n_sim (and the bootstrap sibling) is validated only as positive, and drives an N² × n_sim dense allocation — a latent resource sink: the trust boundary is a constructor keyword (program code), but any caller who exposes hyperparameters to config or a UI turns it into a process-kill path.

Direction. Mirror the max_comb idiom — an overridable ceiling checked at construction — on NormalUncertaintySet and ARCHUncertaintySet.

  • bounds a process-kill sink
  • reuses existing max_comb idiom
  • overridable — no capability lost
  • blast radius: 2 constructors

Top recommendation — close the non-finite data boundary first

Finding 1 is the one remaining silent-wrong-answer path reachable from the primary API with default settings: a single bad tick or zero price flows through prices_to_returns or a direct ReturnsResult into the Gerber-family comparison sinks, where NaN votes are silently masked and the optimiser returns confident weights off a corrupted covariance. The fix is ~4 lines at one choke point the whole pipeline already passes through, and it completes the item the 2026-07-06 review left half-done. Finding 2 is the same fail-open-vs-closed call at the parser and is a natural companion commit.
