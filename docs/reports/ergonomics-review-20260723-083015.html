Ergonomics review — PortfolioOptimisers.jl

2026-07-23 · branch dev

This pass walked the caller-facing surface — optimise, prior, fit/predict, the JuMPOptimiser/MeanRisk/EntropyPoolingPrior constructors, the alias table, and the docstring-generation dicts — looking for friction a caller hits before reading the source. It found a documented call that throws UndefVarError, an exception whose name states the inverse of its cause, two silent corruptions in the generated-doc dicts, and several places where a naming convention breaks or an alias is missing. Prior-review items (Windowed macro, Model-State accessors, custom-objective channel, ucs-triple naming, var_views docstring) are excluded.

1 · The documented happy path throws UndefVarError: predict is never exported

Strong discoverability
src/23_Pipeline/03_Pipeline.jl:833, :859
Before — caller follows the docstring
julia> using PortfolioOptimisers
julia> res = fit(pipe, prices)          # exported — resolves
julia> predict(res, future_prices)      # docstring says predict(res::PipelineResult, ...)
ERROR: UndefVarError: `predict` not defined
After — the pair the docs promise is reachable
julia> res = fit(pipe, prices)
julia> predict(res, future_prices)      # resolves like its sibling fit

Problem. export Pipeline, PipelineResult, fit exports fit but not predict, while the fit_predict docstring at line 833 tells the caller to call predict(res::PipelineResult, data). After using PortfolioOptimisers the documented call fails with UndefVarError unless the caller qualifies StatsAPI.predict — a fact discoverable only by reading source.

Direction. The out-of-sample half of the workflow becomes callable by the name the docs already use, so fit and predict read as one pair.

  • signal: documented name resolves
  • convention holds: fit/predict pair
  • removes UndefVarError trap
  • blast radius: one export line

2 · IsNothingError is thrown when a value is not nothing

Strong errors
src/20_Optimisation/10_JuMPOptimiser.jl:614-615 · src/01_Base.jl:1860
Before — the exception name states the inverse of the cause
# type doc, 01_Base.jl:1860
"Exception type thrown when an argument or value is unexpectedly `nothing`."

# 10_JuMPOptimiser.jl:614 — thrown because sbgt is NOT nothing
@argcheck(isnothing(sbgt),
          IsNothingError("sbgt must be nothing when bgt is a
                          BudgetCostEstimator, got sbgt = $sbgt"))
After — the exception type matches the caller's mistake
# a "must-be-absent" check reads as a conflict, not a nothing-ness error
ConflictingArgumentError("sbgt must be nothing when bgt is a
                          BudgetCostEstimator, got sbgt = $sbgt")

Problem. IsNothingError is documented and named for the case "value is unexpectedly nothing", but this site raises it for the opposite — sbgt must be nothing and isn't. A caller reading the exception type learns the reverse of what actually went wrong; the low-signal type fights the correct message string.

Direction. A "must-be-absent" constraint reports through a type whose name matches it, so the exception type carries the same signal as the message.

  • signal: type matches cause
  • convention: name ≠ inverse of meaning
  • message text already high-signal
  • blast radius: one throw site

Before — duplicate key inside one Dict
const arg_dict = Dict(

Problem. arg_dict defines :pler twice within the same Dict(...); Julia keeps the last, so the intended "Phylogeny estimator or result" description silently never renders. Separately, the docstring example at :22-29 shows display names (cle, pl) that disagree with the authoritative const at :174-179 (clr, ple) from which every rendered docstring is actually derived.   • no silently-dropped description
  • convention: one key, one meaning
  • blast radius: two dict entries
Note. field_dict and ret_dict are all derived from these consts (:729), so a single mistyped key propagates to every docstring that references it — the reach is the whole generated-doc surface, not one page.

4 · sub-group vs sub-grouped — a whole kwarg family split on one silent suffix

Worth exploring naming
src/01_Base.jl:478-499 · src/20_Optimisation/10_JuMPOptimiser.jl:347-368
Before — the "-ed" is the whole distinction
slt · Sub-group long thr. sglt · Sub-grouped long thr. sst · Sub-group short thr. sgst · Sub-grouped short thr. smtx · Sub-group matrix sgmtx · Sub-grouped matrix scard · Sub-group card. sgcarde · Sub-grouped card. est.
After — the two meanings named apart

A caller cannot tell from slt vs sglt which one they want; the docs disambiguate only by the word "grouped". Naming the two axes distinctly — or documenting the concrete difference in one sentence per pair — lets the caller pick without reading source.

Problem. Eight JuMPOptimiser kwargs come in s*/sg* pairs whose only documented difference is "group" vs "grouped". The convention that the caller has learned — s = sub-group — quietly gains a second, near-identical member with no signal separating them. Adjacent to this, card/scard are bare integers while gcarde/sgcarde are estimators, so the e-for-estimator suffix convention also breaks within the same family.

Direction. The two group axes carry names (or one-line docs) that state their concrete difference, and the cardinality quartet suffixes consistently for its type.

  • signal: name states the axis
  • convention: -e suffix holds
  • fewer source-reads to pick
  • blast radius: kwarg family, aliasable

  • convention: one style per surface
  • signal: opt means one thing

  • blast radius: kwarg renames, aliasable

8 · Errors written in the same abbreviations the caller couldn't decode

Worth exploring errors
src/20_Optimisation/09_JuMPConstraints/03_BudgetConstraints.jl:527 · src/03_Preprocessing.jl:384, :403
Before — codes in the message, internal names in the auto-text

"gbgt is over-determined: bgt ($bgt) and sbgt ($sbgt) already pin the

After — expand once, name the caller's kwarg

Problem. The budget error is well-reasoned but written entirely in the bgt/sbgt/gbgt codes the caller already struggled to decode when setting them — it never expands them once. Direction. Error text expands each code the first time it appears and names the caller's kwarg rather than the internal variable.

  • signal: codes expanded once
  • names kwarg, not internal var
  • message decodes itself
  • blast radius: message text only

9 · The minimal optimise call is a three-level ceremony

Speculative defaults
src/20_Optimisation/11_MeanRisk.jl:71, :99 · 14_RiskBudgeting.jl:379 · 10_JuMPOptimiser.jl:337
Before — three required wrappers before any knob
MeanRisk(;
  opt = JuMPOptimiser(;
    slv = Solver(;
      solver = Clarabel.Optimizer)))
# opt required, slv required — every caller pays this
# before touching a single risk/objective knob
After — one solver arg reaches the happy path
MeanRisk(; slv = Clarabel.Optimizer)
# opt defaults to JuMPOptimiser; a bare solver module
# is promoted to Solver(JuMPOptimiser(...)) internally

Problem. The two most-used optimisers make opt::JuMPOptimiser a required kwarg, and JuMPOptimiser in turn makes slv required, so the shortest working call is the nested three-level form the docstring itself shows. A caller pays the full ceremony before reaching any risk or objective choice.

Direction. A caller who only has a solver reaches a working optimisation with one argument; the wrapping still exists but is not mandatory boilerplate.

  • default: opt covers the common case
  • happy path first
  • solver promotion cuts nesting
  • tension: no universal solver default
Why speculative. There is no solver that covers ≥90% of callers, so slv cannot itself get a default — this is about collapsing the nesting, not defaulting the solver. Weigh against the library's explicit-configuration convention before pursuing.

Top recommendation — export predict so the documented workflow resolves

Start with Card 1. It is the highest friction-to-fix ratio on the list: a caller who follows the fit_predict docstring verbatim hits UndefVarError: predict not defined, and the fix is one token on one export line. Every other card improves a name, a default, or a message; this one turns a documented, currently-broken happy path back on. Pair it with Card 2 (the inverted IsNothingError) and Card 3 (the silent doc-dict corruptions) — all three are Strong, low-blast-radius, and independent of the naming/alias discussions.
