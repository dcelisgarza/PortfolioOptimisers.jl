1 · The scorer sign convention is invisible — and the shipped example tunes toward the worst portfolio

Strong errors docs
src/20_Optimisation/02_CrossValidation/09_Base_SearchCrossValidation.jl:45–68 · 10_GridSearchCrossValidation.jl:149 · src/01_Base.jl:541
Before — the contract the caller reads, and the example they copy
“(::CrossValidationSearchScorer)(scores::AbstractMatrix): Returns the index
 of the optimal parameter set.”                       # full interface doc
“`scorer`: Scoring function.”                         # full field doc

julia> (s::MyScore)(X::Matrix{Float64}) =
           argmin(dropdims(mean(X; dims = 1); dims = 1))   # shipped jldoctest
The matrix is pre-signed higher-is-better — sgn = ifelse(bigger_is_better(r), 1, -1), then test_scores[:, i] = sgn * expected_risk(r, predictions). Nothing the caller reads says so. argmin here selects the worst candidate, contradicting the shipped HighestMeanScore (argmax). No error fires.
After — the convention stated where the caller writes the scorer
“`scores` is orientation-normalised: risk-measure scores are negated so that
 **higher is always better**, whatever measure `r` is. A scorer returns the
 index of the column with the best (highest) aggregate score.”

julia> (s::MyScore)(X::Matrix{Float64}) =
           argmax(dropdims(mean(X; dims = 1); dims = 1))   # corrected example

Problem. A caller writing a custom scorer has no signal that the score matrix is orientation-normalised, and the one example they copy silently selects the worst portfolio.

Direction. State the higher-is-better contract in the scorer interface doc and the scorer field doc, and flip the jldoctest to argmax so example and shipped scorers agree.

  • closes a silent-wrong-answer path
  • signal: contract stated at write-site
  • example matches shipped convention
  • blast radius: docstrings only

2 · wn2/wnp/wninf read as norm ceilings but are diversification floors

Strong naming docs
src/20_Optimisation/10_JuMPOptimiser.jl:369–371 · src/01_Base.jl:500–502
What the name signals
wn2   = 5    # “‖w‖₂ ≤ 5”?
wninf = 5    # “‖w‖∞ ≤ 5”?
             # (meaningless when 1ᵀw = 1)
Say it aloud: weight-norm-2 equals five. The value's apparent unit is a norm magnitude.
What the value means
wn2   = 5    # effective assets ≥ 5
             # inv(norm(w,2)^2) >= wn2
wninf = 5    # largest weight ≤ 1/5
             # inv(norm(w,Inf)) >= wninf
The unit is an asset count (a reciprocal-norm floor) — the opposite direction and different units from what the name signals.

Problem. The caller sets a diversification floor believing they set a norm ceiling; the number's magnitude feels wrong in both directions, and the current one-line docs lead with the mechanism (inv(norm(w, 2)^2)) rather than the meaning.

Direction. Keep the names (the nea→wn2 rename was deliberate) and fix the signal in the docs: lead every member's docstring with “diversification floor — minimum effective number of assets” before the formula, give each a worked value (wn2 = 5 ⇒ at least 5 effective assets; wninf = 5 ⇒ max weight 20%), and cross-link the three as one family.

  • signal: units stated before formula
  • convention: family documented as one
  • no rename — zero blast radius
  • worked values anchor magnitudes
Decision callout. The family was renamed nea → wn2 (+ new wnp/wninf) deliberately during the #146 additions. This card does not re-litigate the name — the direction is docstring signal only. If the rename rationale is load-bearing, it deserves a one-paragraph ADR so future reviews stop here.

3 · The budget family's doc depth is inverted: bgt/sbgt get one clause, gbgt/xbgt get paragraphs

Strong docs
src/01_Base.jl:470–473
Before — verbatim, in field order
:bgt  => "`bgt`: Portfolio budget constraint."                      # 4 words
:sbgt => "`sbgt`: Short-sale budget constraint."                    # 4 words
:gbgt => "`gbgt`: Gross budget (leverage) constraint … Unlike `bgt` and `sbgt`
          — which pin the net and gross exposures only *together* — …"   # ~90 words
:xbgt => "`xbgt`: Whether to pin the long/short decomposition exactly. When
          `false` (the default), `lw` and `sw` are upper bounds … so `bgt`,
          `sbgt` and `gbgt` bound the realised exposures rather than pinning
          them — a short budget of `0.3` means *at most* 30% short. …"   # ~130 words
After — parallel structure, semantics stated where first read
:bgt  => "`bgt`: Net budget `1ᵀw`. A number pins it, a `BudgetRange` bounds it.
          By default budgets *bound* the realised exposure (see `xbgt`)."
:sbgt => "`sbgt`: Short-side budget. `sbgt = 0.3` means *at most* 30% short
          unless `xbgt` pins the decomposition. With `bgt`, fixes net + gross
          together; for gross alone see `gbgt`."

Problem. The two keywords every caller reaches for first are documented in four words each — never mentioning BudgetRange or the bound-vs-pin semantics — while the two opt-in keywords that reference them carry the whole explanation, so understanding bgt requires reading xbgt.

Direction. Rebalance: give bgt/sbgt the accepted-value forms and the bound-vs-pin sentence up front, and let gbgt/xbgt reference them instead of the reverse.

  • signal: semantics at first-read keyword
  • convention: family reads as one voice
  • discoverability: BudgetRange surfaced
  • blast radius: 4 docstrings
Carry-over note. The wider “~50 two-to-four-letter JuMPOptimiser keywords” finding from July 7 was mitigated with the user-guide prose on the abbreviation scheme rather than long-form aliases; it is not re-raised here as a separate card.

4 · RandomisedSearchCrossValidation rejects the Symbol keys its own docstring promises

Strong errors
src/20_Optimisation/02_CrossValidation/09_Base_SearchCrossValidation.jl:240–265 (vs GridSearch :93–109, :188–205)
Before — sibling drift
# GridSearchCrossValidation: keys validated to GSCVKey (String|Symbol|Integer)
GridSearchCrossValidation(Dict(:alpha => [0.1, 0.2]))        # works

# RandomisedSearchCrossValidation: signature hard-codes AbstractString keys,
# but its Validation section still says:
#   “All keys in `p` must be of type `GSCVKey` (i.e. `String`, `Symbol`, or `Integer`).”
RandomisedSearchCrossValidation(Dict(:alpha => Uniform(0, 1)))
ERROR: MethodError: no method matching RandomisedSearchCrossValidation(::Dict{Symbol, …})
After — one convention across both siblings
# Same key contract as GridSearch: loose signature + @argcheck against GSCVKey,
# so a caller can swap Grid ↔ Randomised without re-typing their keys.
RandomisedSearchCrossValidation(Dict(:alpha => Uniform(0, 1)))   # works

Problem. A key type that works verbatim in GridSearchCrossValidation throws a MethodError in its randomised sibling, while the copied docstring promises it is valid — the convention teaches a lie exactly when the caller swaps schemes.

Direction. Widen the randomised constructor's key type to match GridSearch's GSCVKey validation (or, if strings-only is deliberate, fix the docstring and raise a typed error naming the fix).

  • convention: siblings accept same keys
  • docstring matches behaviour again
  • blast radius: 1 constructor
  • Grid ↔ Randomised swap is free

7 · The raw eps keyword sits beside the calibration built to replace it

Speculative defaults discoverability
src/14_UncertaintySets/05_L1UncertaintySets.jl:118–130 (raw eps) · :20–30 (the trap, documented)
Trap path — equally reachable
L1UncertaintySet(; eps = 0.5)
# only guard: isfinite(eps) && eps >= 0
# but ε has no natural scale: ~1e-3 on
# daily returns, ~250× larger annualised
# — 0.5 on daily data ⇒ 1/N, silently
Calibrated path — the quantity a caller can reason about
CharacteristicUncertaintySet(;
    method = ActiveAssetsUncertaintyAlgorithm(;
        active = 0.2))   # “hold ~a quintile”
# inverts the closed forms to find ε

Problem. The scale trap is documented and the calibration exists, but the unscaled eps keyword remains equally reachable and equally prominent, so the trap path is one autocomplete away from the safe one and mis-scales silently with data frequency.

Direction. Steer, don't remove: put the calibrated form first in the eps docstring and both constructors' examples, and cross-link ActiveAssetsUncertaintyAlgorithm from eps itself (“if you are guessing this number, use the calibration”); a magnitude heuristic warning is possible but likely too clever.

  • discoverability: safe path first
  • default stays — expert escape hatch
  • blast radius: docstrings only
  • memory-flagged gotcha closed at source
ADR 0032. Quintile portfolios are deliberately a recipe over this set, not an optimiser type — no QuintilePortfolio wrapper is proposed here; the direction is purely which constructor form the docs lead with.
