1 · Non-finite returns pass the ReturnsResult trust boundary

src/03_Preprocessing.jl:362–414 (inner constructor); contrast src/03_Preprocessing.jl:982 (prices_to_returns sanitizes)

Problem. The ReturnsResult inner constructor is the boundary where untrusted return data (CSV/spreadsheet imports) enters the library, and it validates everything except the data itself: names get allunique, dimensions get DimensionMismatch, and the optional iv/ivpa fields get full finiteness checks — but X, F, and B are never checked with isfinite, so a NaN from a blank spreadsheet cell flows into moments, distances, and clustering and can surface as a silently wrong portfolio.

Direction. Extend the constructor's existing assert_*_finite_val discipline (already applied to iv/ivpa) to X/F/B — the check is O(n) once, against workflows that cost seconds. No documented workflow carries NaN at returns level: imputation is prices-level (inside prices_to_returns, which already maps NaN→missing and drops), so the unguarded path is only direct construction — exactly the untrusted-import path.

  • fail closed at the boundary
  • closes the silent-wrong-weights path
  • symmetric with iv/ivpa checks
  • blast radius: one constructor

2 · Benchmark names skip the uniqueness check assets and factors get

src/03_Preprocessing.jl:366–373 (nx, nf checked) vs 376–380 (nb not)

Problem. The same constructor enforces allunique(nx) and allunique(nf) but places no uniqueness check on nb, so the three name vectors have two different validation contracts at one boundary.

BEFORE — asymmetric contract
nx dupes → ArgumentError   ✓
nf dupes → ArgumentError   ✓
nb = ["SPY","SPY"] → accepted ✗
AFTER — one contract
if !isnothing(nb)
    @argcheck(allunique(nb), ...)
end

Direction. Add the missing allunique(nb) branch. Severity is Low because nb is only carried and sub-selected by Views (src/03_Preprocessing.jl:574,586) and CV prediction assembly — it is never resolved name→index for constraint generation — so today duplicates confuse labels, not results. The cost of the gap is boundary uniformity: a future consumer that does resolve benchmark names inherits an unvalidated field.

  • one validation contract per boundary
  • closes gap before a consumer exists
  • blast radius: three lines
Low Worth exploring input-validation LATENT — needs config-fed integer

3 · n_sim has a floor but no ceiling — unbounded O(N²·n_sim) allocation

src/14_UncertaintySets/04_BootstrapUncertaintySets.jl:284 (check), 338–339, 420, 735, 886 (allocations)

Problem. ARCHUncertaintySet validates n_sim > 0 and nothing else; the fit then allocates Array(N, N, n_sim) and Matrix(N², n_sim), so an n_sim fed from a config file drives allocation with no cap and no warning — an OOM process kill. The library already has the exact countermeasure pattern one directory over: CombinatorialCrossValidation fails closed at max_comb = 100_000 with a message explaining sensible ranges.

BEFORE — floor only
# config file: {"n_sim": 100000000000}
ue = ARCHUncertaintySet(; n_sim =
        100_000_000_000)   # ✓ constructs
sigma_ucs(ue, X)  # 50 assets →
# Array(50, 50, 1e11) ≈ 2 PB request
# → OOM kill or swap death
AFTER — the max_comb pattern
# mirror 03_Combinatorial.jl:66-72
ARCHUncertaintySet(; n_sim,
    max_sim = 1_000_000)
@argcheck(n_sim <= max_sim,
  ArgumentError("n_sim = $n_sim exceeds
  max_sim = $max_sim; typical values
  are 10^3–10^4 ..."))

Direction. Mirror the max_comb keyword-with-default idiom so machine-authored large runs stay possible (raise the cap explicitly) while a fat-fingered or hostile config integer fails closed at construction. This was the "n_sim ceiling" item left open by the 2026-07-06 review; it has no accepted-risk ADR, so it stands. The rng/bootstrap path itself is clean — no other uncapped bootstrap counts found.

  • process-kill path fails closed
  • reuses existing max_comb idiom
  • blast radius: one constructor
Low Speculative input-validation LATENT — developer-authored input

4 · Search-CV allocators materialise eagerly with no bound

src/20_Optimisation/02_CrossValidation/10_GridSearchCrossValidation.jl:22,27 · 11_RandomisedSearchCrossValidation.jl:29

Problem. lens_val_grid fully materialises vec(collect(Iterators.product(...))) — the product of every candidate-value list — and make_p_grid calls rand(rng, dist, n_iter) with no ceiling on n_iter. Ten parameters × ten values each is 10¹⁰ tuples allocated before the first fit runs.

BEFORE — eager, unbounded
vals = vec(collect(
  Iterators.product(lists...)))
# |grid| = ∏ length(list) — allocated
# up front, silently
AFTER — cheap pre-check
n = prod(length, lists)  # O(k) ints
n <= max_grid || throw(
  ArgumentError("grid has $n points,
  exceeds max_grid = $max_grid"))

Direction. A prod(length, ...) pre-check (integers only, before any tuple exists) matched against a max_grid default would fail closed for the cost of one multiply. Marked Speculative because the ADR 0025 amendment already classifies search-CV specs as developer-authored code, not config/UI input — the same trust class as parse_lens keys — so this is boundary uniformity, not a reachable exploit. Reasonable to decline on the same grounds as ADR 0027's findings 1/3.

  • fail closed before allocation
  • consistent with max_comb idiom
  • latent: developer-authored trust class

Top recommendation

Close Finding 1 first: add isfinite validation for X/F/B in the ReturnsResult inner constructor. It is the only finding reachable from the documented public API with default settings, and its worst case is the worst category — a silently wrong portfolio through the solver-free hierarchical paths, where nothing downstream ever errors. The fix is a three-branch extension of the finiteness discipline the same constructor already applies to iv/ivpa, it cannot break any documented workflow (imputation is prices-level, and prices_to_returns already sanitizes), and its blast radius is one constructor plus a regression test. Findings 2 and 3 ride along naturally in the same change; Finding 4 is a judgement call the ADR record arguably already answers.
