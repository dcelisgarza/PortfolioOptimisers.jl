1 · Collapse the nine Range twins onto the generic loss/gain interface

Strong in-process
src/19_RiskMeasures/06_XatRisk.jl:424 · 07_ConditionalXatRisk.jl:360,475 · 08_EntropicXatRisk.jl:234 · 09_RelativisticXatRisk.jl:330 · 19_PowerNormXatRisk.jl:273 · 10_OWARiskMeasures.jl:1285,1426,1781
src/20_Optimisation/20_RiskMeasureConstraints/07_ConditionalXatRiskConstraints.jl (117 KB — largest file in src/) + 06, 08, 09, 10, 19
the deep shape already exists: 21_GenericValueatRiskRange.jl:115-166 + 21_GenericValueatRiskRangeConstraints.jl (~12 lines)

Problem. A Range is one concept — lower-tail loss + upper-tail loss — but nine bespoke Range measures re-encode it inline. Each struct repeats the settings/alpha/beta/w field block and validation; each emitter hand-mirrors a _l/_h copy of the single-tail formulation applied to net_X and -net_X. Meanwhile GenericValueatRiskRange proves the whole template is a 4-line functor plus a 12-line emitter that calls set_risk_constraints! twice through the existing loss= keyword interface.

Before — nine shallow twins
CVaR Range
var_l/z_l ⧉ var_h/z_h
EVaR Range
_l ⧉ _h mirror
RVaR Range
_l ⧉ _h mirror
PNVaR Range
_l ⧉ _h mirror
VaR Range
_l ⧉ _h mirror
DR-CVaR + 3×OWA
_l ⧉ _h mirror

Each box: full field block + validation + ~41–85-line mirrored emitter. One tail-formulation bug = nine fixes.

After — one deep module, nine thin adapters
CVaR EVaR RVaR PNVaR OWA ×3
Generic Range
loss(x) + gain(−x)
emitter: set_risk_constraints! ×2 via loss= keyword — the tail formulation lives once, in the single-tail measure

The interface stays what callers already know: a Range measure with alpha/beta constructors, now adapters over loss/gain sub-measures.

Solution. Re-express each bespoke Range as the generic shape: the struct holds loss/gain sub-measures (constructors keep today's alpha/beta ergonomics), the emitter becomes the twofold delegation. First step — the test seam: test_03b builds models solver-free through assemble_jump_model! but has zero Range coverage; pin each Range's emitted Model State keys there before the collapse, so the refactor is verified through the interface, not through end-to-end solves in test_18*.

  • Leverage: one two-tail implementation, nine call sites
  • Locality: tail bugs fix once, fixed everywhere
  • The 117 KB emitter file shrinks severalfold
  • Interface is the test surface: solver-free per-measure tests

2 · One declaration per Risk Measure — close the residual sync surface

Worth exploring in-process
src/19_RiskMeasures/ (risk_input_kind ×54, supports_precomputed_returns ×8, functors, @propagatable ×39 + @pprop ×39 …) · src/20_Optimisation/20_RiskMeasureConstraints/ (set_risk_constraints! ×55) · test/test_09c_risk_input_kind.jl · test/test_29

Problem. The interface a new Risk Measure must satisfy is not written anywhere — it is implied by four method families spread over two parallel-numbered trees, and drift is caught only retroactively by completeness tests. ADRs 0013/0018 already closed the factory/view and compatibility legs by deriving them; the input-kind trait, the precomputed-returns contract, the functor, and the propagation annotations are still hand-synced per measure.

Before — a new measure crosses 5 bands in 2 trees
struct + functor 19_RiskMeasures/NN_·.jl
risk_input_kind — 54 hand defs scattered
precomputed-returns contract — 8 defs + forwarders
@propagatable / @pprop / @cprop annotations
set_risk_constraints! — 55 defs 20_·Constraints/NN_·.jl

Forget a band → test_09c/test_29 fail after the fact. No checklist exists in src/ or docs/.

After — declaration emits the traits
@risk_measure declaration
kind = NetReturns · precomputed = true · fields…
emits: input-kind trait · precomputed contract · functor stub · propagation tags — in sync by construction
set_risk_constraints! stays hand-written — it is the real implementation, not boilerplate

test_09c/test_29 stop policing and start documenting.

Solution. A single declaration seam (macro or trait-carrying supertype, in the spirit of ADR 0018) from which input kind, the precomputed-returns contract, the functor stub, and propagation tags are derived together. The JuMP emitter remains a separate hand-written method — that one is depth, not drift.

  • Interface shrinks: one declaration, four derived facts
  • Locality: adding a measure touches 2 files, not 5 bands
  • Drift becomes unrepresentable, not just detected

4 · Let the MIP indicator bundle live in Model State, not in return values

Worth exploring in-process
src/20_Optimisation/·/01_MIPIndicators.jl (bundle + accessors) · 06_IntegerPhylogenyConstraints.jl:6-10 (the hand-off comment) · test/test_28_seam_lock.jl:20-32 (regex police)

Problem. ADR 0033's typed indicator bundle is the right interface, but the held indicator's identity still leaks across the seam: each builder registers it under a different raw key (:ib vs :ilb/:isb), so emitters thread the bundle back as a return value and pass it by hand — and the "reach it only through the bundle" rule is enforced by a source-regex seam-lock test, not by the interface. The abstraction is policed externally instead of being unrepresentable to violate.

Before — hand-threaded hand-off
sequenceDiagram
  participant M as set_mip_constraints!
  participant B as builder (long-only / long-short / sign)
  participant P as phylogeny / card emitter
  M->>B: run_mip_builder!
  B-->>M: bundle (return value)
  M-->>P: bundle passed by hand
  Note over P: model[:ib] would KeyError on
short+threshold models —
regex test forbids raw keys
After — bundle is Model State
sequenceDiagram
  participant M as set_mip_constraints!
  participant S as Model State
  participant P as any late emitter
  M->>S: register bundle under one canonical entry
  P->>S: held(model) accessor
  Note over S: raw keys become private
implementation of the bundle —
seam-lock test shrinks to a doc

Solution. Register the bundle itself as a first-class Model State entry (the same named-interface pattern ADR 0004 gave the rest of Model State); emitters that run late — integer phylogeny applied after the fact — read it back through the typed accessors instead of receiving it hand-threaded. The per-builder raw keys become private to the bundle's implementation.

Builds on ADR 0033 — this completes the bundle seam it introduced; the locked key list stays, it just stops being the only enforcement.
  • Seam enforced by types, not by regex
  • Locality: indicator identity decided in one module
  • Late emitters stop needing threaded arguments

5 · The TimeDependent widening is re-encoded ~203 times

Speculative in-process
src/20_Optimisation/01_Base_Optimisation.jl (2,366 lines — resolve/bind core) · Union{…,TimeDependent} widenings ×~203 across 12 files · JuMPOptimiser: 41 fields

Problem. The schedule machinery itself is deep and concentrated — binding, entitlement, and resolution live in one module. The friction is the interface's footprint: "this field may carry a schedule" is a single fact per field, re-declared as a widened constructor signature ~203 times. If the schedule contract ever changes shape, the change amplifies across 12 files.

Interface vs implementation mass
widened sigs ×203
resolve/bind core ×1

The declared surface outweighs the one implementation it points at.

⚠ Deliberate per ADR 0030

The widened signatures are the admissibility table — they document, per input, that it is problem definition rather than execution control. Any consolidation (e.g. a per-field admissibility declaration the constructors derive from) must preserve that documentation role. Not worth touching unless the schedule contract changes again; recorded here so the next contract change triggers the consolidation rather than a 203-site edit.

  • Watch item: act on next contract change
  • Core module is already deep — leave it

6 · Pipeline slot routing rests on two hand-synced tables

Speculative in-process
src/23_Pipeline/01_Base_Pipeline.jl:81-99 (PIPELINE_SLOT_INVALIDATIONS) · :194-258 (pipe_reads/pipe_writes by family) · :38-60 (uncertainty mu/sigma target intent)

Problem. Which Pipeline Context slot a step family reads/writes, and which downstream slots a write invalidates, are two separate hand-maintained tables that must track the estimator taxonomy — plus a third wrinkle where an uncertainty Result cannot say whether it targets mean or covariance, so intent rides on the step wrapper. Adding a new step family means remembering all three places.

Before — three places per family
pipe_reads(::Family) / pipe_writes(::Family)
PIPELINE_SLOT_INVALIDATIONS[slot]
PipelineStep wrapper: target = :mu/:sigma
After — one slot declaration
slot contract per family
reads · writes · invalidates — one declaration; the invalidation table derived from the slot dependency order

Deletion test: the invalidation table restates the workflow order (data → prior → … → weights) that the slot list already encodes.

  • Locality: one declaration per step family
  • Invalidation derived, not hand-mirrored
  • Low urgency: tables are small and tested today

Checked and healthy — no action

  • • RandomisedSearchCrossValidation delegating to grid search: thin, but the deletion test says merging would move complexity, not concentrate it. The 3-way score-shape dispatch (contiguous / combinatorial / randomised) is essential variance.
  • • Equation parser (02_LinearConstraintGeneration.jl, 1,679 lines): large but deep — one interface, hardened by ADRs 0025–0027.
  • • dims/square-assert preamble: down to 17 scattered sites; retired as a candidate.
  • • Coverage gaps worth noting (not architecture): 04–07 matrix processing, 21_ExpectedReturns.jl, and 25_Aliases.jl have no dedicated test files — only transitive coverage.
Top recommendation

Collapse the Range twins — and land the solver-free extraction tests first

Third review in a row where this is the largest concentration on offer, and the evidence has only hardened: the deep shape already ships (GenericValueatRiskRange, ~1/7 the size), the duplication centres on the single biggest file in src/, and the missing test seam (test_03b has zero Range coverage) is both the prerequisite and a win in its own right. Candidate 2 becomes markedly cheaper afterwards — nine fewer bespoke measures to declare.
