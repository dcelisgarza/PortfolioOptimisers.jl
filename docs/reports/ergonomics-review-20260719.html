1 · EntropyPoolingPrior's own docs say var_views is variance — it is Value-at-Risk

Strong docs
src/13_Prior/10_EntropyPoolingPrior.jl:439, 487, 1034, 1060, 1064 · src/01_Base.jl:337
Before — what the docstrings actually say

:439  "It supports moment and view constraints (mean, variance, CVaR, covariance, skewness, kurtosis, correlation)…" — VaR never mentioned

:1034 "…does nothing when variance view constraints (`var_views`) are not provided…"

:1060 "`ep_var_views!` parses variance (VaR) view equations…"

:1064 "- `var_views`: Variance (VaR) view constraints."

After — one convention, stated everywhere

:439  "…view constraints (mean, Value-at-Risk, CVaR, covariance, skewness, kurtosis, correlation)…"

Warning box on the estimator docstring: "`var_views` constrains Value-at-Risk, not variance. Variance views go through `sigma_views`."

:1034/:1060/:1064 scrubbed of every "variance (VaR)" / "variance view" phrasing.

var_views → caller reads "variance" ✗
var_views → Value-at-Risk (5% tail) ✓
sigma_views → variance ✓
cvar_views → Conditional VaR ✓

Problem. A caller writing var_views = LinearConstraintEstimator(; val = ["A == 0.04"]) to pin asset A's variance silently constrains its 5% Value-at-Risk instead — the intro sentence lists "variance" as supported, never says VaR, and three internal docstrings repeat the wrong reading; the one correct parenthetical sits far down in an operator-comparison subsection.

Direction. Docs-only fix: correct the intro sentence, add a warning box pairing var_views→VaR with sigma_views→variance, and scrub the three "variance (VaR)" internal docstrings; optionally add a value_at_risk_views alias kwarg rather than renaming (~30 uses of var_views in the file).

  • kills a silent-wrong-number path
  • zero blast radius (docs-only)
  • signal: name glossed at every mention
  • convention: matches CONTEXT.md entry
Known gotcha, still open. This trap is already recorded in the project's working notes ("entropy-pooling view naming") — the docstrings were never brought in line with it.

2 · optimise(mr, rd_test) silently ignores rd_test when pe holds a prior result

Strong errors
src/13_Prior/01_Base_Prior.jl:257 · src/20_Optimisation/11_MeanRisk.jl:681 · user_guide/02_Optimisers.jl:54, 93
Before — the silent path
opt = JuMPOptimiser(; pe = pr,  # fitted prior RESULT
                      slv = slv)
mr  = MeanRisk(; opt = opt)

res = optimise(mr, rd_test)
# rd_test is silently ignored:
# prior(pr::AbstractPriorResult,
#       args...; kwargs...) = pr
# ⇒ "out-of-sample" run is in-sample
After — the ignored argument announces itself
res = optimise(mr, rd_test)
┌ Warning: `pe` is a fitted prior result,
│ so the returns passed to `optimise` are
│ ignored. Pass a prior *estimator*
│ (e.g. pe = EmpiricalPrior()) to have
│ the prior fitted on these returns.
└ @ PortfolioOptimisers

Problem. The same call shape means two different things depending on whether pe holds an estimator or a result — the estimator path consumes rd, the result path is a pass-through — and the ignored-argument case gives the caller no signal, so an out-of-sample evaluation silently becomes an in-sample one.

Direction. When pe is a prior result and the positional rd is non-empty, warn (or error) that rd is ignored; state the precedence rule once on the JuMPOptimiser and optimise docstrings. The estimator-vs-precomputed distinction is already documented for cross-validation — this closes the same gap on the plain-solve path.

  • kills a silent in-sample evaluation
  • signal: precedence stated at call site
  • zero blast radius (additive warning)
  • convention: matches CV's estimator rule

Problem. Four fallback methods on high-traffic entry points (fit_preprocessing, apply_preprocessing, select_assets, port_opt_view) answer the most common caller mistake — passing the wrong kind of object — with "implement this method", handing an everyday caller the extension author's mental model.

Direction. Lead each message with what was expected and what was received; keep the "implement …" sentence as a trailing extension-author note (or move it entirely into the abstract type's docstring, where extension guidance already lives).

  • signal: error names caller's mistake
  • fix suggestion in the message

Problem.
