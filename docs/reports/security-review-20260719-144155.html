Security review — PortfolioOptimisers.jl

2026-07-19 · branch dev

Third hardening pass. The injection surface closed by ADRs 0025–0027 (enumerated parser allowlist, recursion caps, info-leak-safe diagnostic builders) is holding: both Meta.parse sites are capped, no eval/subprocess/network/deserialization sink exists in src/ or ext/, and every asset/group/view name-resolution boundary routes through the shared builders. The pass found the resource half of the trust perimeter still open — untrusted config integers sizing allocations with no ceiling — plus a caller-state contamination path in RNG seeding, three small boundary-discipline drifts in newer code (Pipeline, search-CV), and one High carried over unfixed from the 2026-07-07 pass: non-finite returns reaching the comparison-based moment estimators and producing a silent wrong answer. All six are now implemented and verified — three of them re-scoped away from the direction written here, each course correction recorded on its card.

Problem. The comparison-based covariance estimators (GerberCovariance, SmythBrobyCovariance) count co-movement with X .>= sd / X .<= -sd, where a NaN compares false in both directions and is silently absorbed as "no move" — so untrusted returns reach the sink and the blast radius is a silent wrong result (the worst category), reachable from the exported API with defaults.

Direction taken. A shared assert_all_finite helper (info-leak-safe: count + first index, never the values) fails closed at all four cor/cov entry points. The check was deliberately not placed on ReturnsResult: a NaN-carrying returns matrix is a documented, first-class input to the Asset Selector / MissingDataFilter family (see CompleteAssetSelector, whose own doctest builds one) — cleaning NaN is a supported stage, so the boundary belongs at the estimators that cannot tolerate it, not at the data type.

  • closes a silent-wrong-answer path
  • fails closed at the sink, not the data
  • selector/MissingDataFilter workflow intact
  • test_08 moments 515/515 green
Course correction: the review first proposed a ReturnsResult-constructor check. Tracing revealed CompleteAssetSelector's doctest deliberately feeds a NaN-carrying returns matrix — NaN is a supported selector input, not a corruption — so a boundary check would have been a regression. The fix was re-scoped to the comparison sinks. Standard estimators are left alone: they propagate NaN loudly through posdef/Cholesky rather than masking it.

Problem. Three exported constructors accept an untrusted integer that directly multiplies an allocation or a full-optimisation loop with only a lower bound — n_sim in both NormalUncertaintySet (Wishart/MvNormal draws of N²·n_sim) and ARCHUncertaintySet (N²·n_sim bootstrap array), and n_subsets in SubsetResampling (whose only ceiling, binomial(N, subset_size), is astronomically large for realistic universes) — so a config-supplied value reaches the allocation sink unchecked and the blast radius is a process kill (OOM / compute exhaustion), reachable from the exported API.

Direction taken. A new ResourceLimits / RESOURCE_LIMITS ScopedConfig mirrors the EQUATION_LIMITS idiom (ADR 0027) exactly — struct with a positive-field inner constructor, set_resource_limits! global setter, with_resource_limits scoped override, and load-time "max_samples"/"max_subsets" preferences that fail closed. A shared assert_resource_cap helper names both the rejected field and the knob that raises it. Two fields rather than one: max_samples (default 1_000_000) bounds memory, while max_subsets (default 100_000) bounds compute — every subset runs a full inner optimisation — so they warrant different ceilings and different messages.

Placement, re-scoped. n_sim is a plain Integer field, so its check sits in both uncertainty-set inner constructors as the review proposed. n_subsets could not: it is a TimeDependent-capable field that may hold a schedule or a callable, and its concrete value only appears at optimise time via get_n_subsets. The cap therefore lives at that resolution seam — which also turned out to cover a sink the review missed, MultipleRandomised, since both it and SubsetResampling route through the same function.

3 · seed reseeds the caller's task-global RNG in place — FIXED

Strong Medium least-privilege Implemented 2026-07-20
src/14_UncertaintySets/03_NormalUncertaintySets.jl:357,490,587 · 04_BootstrapUncertaintySets.jl:341,382,422 · src/20_Optimisation/03_NaiveOptimisation.jl:766 · 02_CrossValidation/05_MultipleRandomised.jl:345 · 11_RandomisedSearchCrossValidation.jl:81 · src/23_Pipeline/05_SearchCrossValidation.jl:348 · src/11_Phylogeny/05_NonHierarchicalClustering.jl:107
Before — seeding a shared object the caller also owns
ue = NormalUncertaintySet(; seed = 42)  # rng defaults to Random.default_rng()
ucs(ue, X)    # Random.seed!(ue.rng, 42) → reseeds the caller's task-local RNG
rand()       # caller's "random" stream is now deterministic from 42 — silently
After — a seed buys a private stream, not the shared one
# resolve_rng(rng, seed) = isnothing(seed) ? rng : Random.seed!(copy(rng), seed)
ucs(ue, X)    # draws from a private, reseeded copy; reproducible, self-contained
rand()       # caller's stream untouched

Problem. Every seeded estimator defaults rng = Random.default_rng() and, when seed is set, calls Random.seed!(ue.rng, ue.seed) — so with default settings the library mutates the caller's task-global RNG state, holding more capability than reproducibility needs and silently derandomising every downstream consumer of the global stream (Monte-Carlo code, other libraries, test frameworks); reachable from the exported API with defaults.

Direction taken. One resolve_rng(rng, seed) helper in 01_Base.jl, applied at all 13 draw sites: a supplied seed yields a private, reseeded copy of rng; with no seed, rng is used as-is; the shared default rng is never seeded in place. The kmeans site additionally now threads the resolved rng into Clustering.kmeans(...; rng) — it previously passed nothing and worked only by the side effect on the global stream, so it was silently unreproducible for any caller-supplied rng.

  • least privilege: no global mutation
  • blast radius: one helper, 13 sites
  • byte-identical draws — no CSV regeneration
  • closes silent caller-state contamination
Course correction: the review proposed isnothing(seed) ? rng : Random.Xoshiro(seed). That discards the caller's generator type — and the test suite deliberately pairs rng = StableRNG(...) with a seed so the recorded streams stay portable across Julia versions. The literal helper broke test_10 (2 errored) on exactly those cases. Re-scoped to Random.seed!(copy(rng), seed), which preserves both the object (never mutated) and the type: verified that seed!(copy(default_rng()), s) ≡ Xoshiro(s) and that a copied-then-seeded StableRNG reproduces the old in-place stream exactly — so the predicted CSV regeneration turned out to be unnecessary.
Second correction — shared-stream sites: a naive per-call swap is not always sufficient. split(::MultipleRandomised, …) has two draw sites: sample_unique_assets for the asset subsets, and rand(rng, 1:(T - window_size)) for each path's window offset. The in-place seed! inside the former used to seed the struct's rng, so the window offsets silently inherited the seeded stream. Routing only the first through resolve_rng left the window draws unseeded and broke 2 test_24 assertions on exact fold ranges. Fixed by resolving once at the top of split and passing the resolved rng down with no seed — one stream serving both sites, reproducing the old sequence exactly. Lesson for finding 2 and any future rng work: when replacing in-place seeding, check for downstream consumers of the same rng object, not just the call being edited.
Verification (complete): test_10 106/106, test_13 phylogeny 123/123 (kmeans), test_14 naive 10/10, test_21 subset resampling 11/11, test_24 cross-validation 211/211, test_34 pipeline search-CV 51/51 — all green, zero CSV changes. Caller-contamination closed directly: interposing a seeded ucs call between two rand() draws leaves the global stream bit-identical to an uninterrupted run.

4 · min_score accepts any Real — a non-positive value turns "did you mean?" into a universe-name oracle — FIXED

Strong Low info-leak Implemented 2026-07-19
src/01_Base.jl:1267 (StringDistanceConfig) · :1300-1303 (set_string_distance!) · :1597-1601 (apply_preferences!)

Problem. The preference path and set_string_distance! validate min_score only as a non-Bool Real, so a zero or negative value silently removes the threshold that keeps did_you_mean from echoing a real asset name for arbitrary probes — undoing the ADR 0026 "never interpolate the universe" discipline one name per warning; latent (requires a bad project preference or setter call), not reachable with defaults.

Direction taken. A new StringDistanceConfig inner constructor enforces min_score > 0 (and rejects NaN), so every route in — set_string_distance!, with_string_distance, load-time preference — fails closed at construction, matching the EquationLimits pattern. The > 1 disable sentinel stays legal.

  • fails closed at config load
  • one constructor guards three routes
  • preserves >1 disable sentinel
  • keeps ADR 0026 leak posture intact
ADR 0026: the suggestion threshold is what makes the fuzzy hint safe to append to the info-leak-safe builders. This finding closes the one configuration that defeats it.
Course correction: the review's problem statement said "a zero or negative value", but its direction proposed min_score >= 0 — which permits exactly the zero case it had just called out. Probing the actual behaviour settled it: findnearest never suggests a candidate scoring exactly 0, so a hopeless probe leaks nothing at any threshold, but min_score = 0 and min_score = -1 are otherwise identical — both admit every candidate with some nonzero similarity ("XPPL" → AAPL at 0.5). Zero is therefore as leaky as negative, and the guard shipped as > 0. The documented range narrows from [0, 1] to (0, 1]; nothing in the repo set it to zero.

All findings closed — what the pass actually taught

All six findings are implemented and verified. The comparison-based estimators fail closed on non-finite returns; seeded estimators draw from a private reseeded copy instead of mutating the caller's task-global RNG; the sampling integers are capped by RESOURCE_LIMITS; and the three boundary-discipline drifts are on the shared builder shape.

The durable lesson is that three of the six shipped somewhere other than where this report pointed, and in each case tracing the code beat trusting the write-up: the ReturnsResult check would have broken a documented NaN-carrying selector input (finding 1); the Xoshiro(seed) helper discarded the caller's generator type and broke StableRNG-seeded tests (finding 3); the n_subsets constructor check could not see a schedule's resolved value, and moving it to get_n_subsets caught a fourth sink the report never listed (finding 2). Finding 4's stated direction even contradicted its own problem statement. Treat the cards as leads, not specifications.
