Security review — PortfolioOptimisers.jl

2026-07-19 · branch dev

Third hardening pass. The injection surface closed by ADRs 0025–0027 (enumerated parser allowlist, recursion caps, info-leak-safe diagnostic builders) is holding: both Meta.parse sites are capped, no eval/subprocess/network/deserialization sink exists in src/ or ext/, and every asset/group/view name-resolution boundary routes through the shared builders. What remains is the resource half of the trust perimeter — untrusted config integers sizing allocations with no ceiling — plus a caller-state contamination path in RNG seeding and three small boundary-discipline drifts in newer code (Pipeline, search-CV). One High carries over unfixed from the 2026-07-07 pass: non-finite returns cross the ReturnsResult boundary unchecked and produce a silent wrong answer in comparison-based moment estimators.

Problem. Three exported constructors accept an untrusted integer that directly multiplies an allocation or a full-optimisation loop with only a lower bound — n_sim in both NormalUncertaintySet (Wishart/MvNormal draws of N²·n_sim) and ARCHUncertaintySet (N²·n_sim bootstrap array), and n_subsets in SubsetResampling (whose only ceiling, binomial(N, subset_size), is astronomically large for realistic universes) — so a config-supplied value reaches the allocation sink unchecked and the blast radius is a process kill (OOM / compute exhaustion), reachable from the exported API.

Direction. Add a runtime-overridable resource cap mirroring the EQUATION_LIMITS idiom (ADR 0027) — conservative default far above legitimate use, checked at the constructor trust boundary, raised explicitly via a setter by callers with genuinely large machine-authored runs.

  • fails closed on absurd integers
  • blast radius: OOM kill → typed error
  • reuses proven EQUATION_LIMITS idiom
  • one config, three sinks closed
Prior review: the n_sim ceiling was an explicitly open item of the 2026-07-06 pass; no ADR accepts the risk. The sibling combinatorial-CV explosion was since closed with a hard max_comb @argcheck — this finding extends the same treatment to the remaining unbounded integers.

3 · seed reseeds the caller's task-global RNG in place

Strong Medium least-privilege
src/14_UncertaintySets/03_NormalUncertaintySets.jl:357,490,587 · 04_BootstrapUncertaintySets.jl:341,382,422 · src/20_Optimisation/03_NaiveOptimisation.jl:766 · 02_CrossValidation/05_MultipleRandomised.jl:345 · 11_RandomisedSearchCrossValidation.jl:81 · src/23_Pipeline/05_SearchCrossValidation.jl:348 · src/11_Phylogeny/05_NonHierarchicalClustering.jl:107
Before — seeding a shared object the caller also owns
ue = NormalUncertaintySet(; seed = 42)  # rng defaults to Random.default_rng()
ucs(ue, X)    # Random.seed!(ue.rng, 42) → reseeds the caller's task-local RNG
rand()       # caller's "random" stream is now deterministic from 42 — silently
After — a seed buys a private stream, not the shared one
# resolve_rng(rng, seed) = isnothing(seed) ? rng : Random.Xoshiro(seed)
ucs(ue, X)    # draws from a private Xoshiro(42); reproducible, self-contained
rand()       # caller's stream untouched

Problem. Every seeded estimator defaults rng = Random.default_rng() and, when seed is set, calls Random.seed!(ue.rng, ue.seed) — so with default settings the library mutates the caller's task-global RNG state, holding more capability than reproducibility needs and silently derandomising every downstream consumer of the global stream (Monte-Carlo code, other libraries, test frameworks); reachable from the exported API with defaults.

Direction. One resolve_rng(rng, seed) helper at the draw sites: a supplied seed yields a private Xoshiro(seed); an explicitly supplied rng is used as-is; the shared default rng is never seeded in place.

  • least privilege: no global mutation
  • blast radius: one helper, ~12 sites
  • reproducibility preserved per estimator
  • closes silent caller-state contamination
Behaviour note: repeated fits of the same seeded estimator already restart the stream (that is what the in-place seed! is for), so a private per-call rng preserves the observable contract — only the side effect on the caller's stream disappears. Existing test CSVs seeded through the global rng may need regenerating; worth checking before committing to the swap.

4 · min_score accepts any Real — a non-positive value turns "did you mean?" into a universe-name oracle

Strong Low info-leak
src/01_Base.jl:1267 (StringDistanceConfig) · :1300-1303 (set_string_distance!) · :1597-1601 (apply_preferences!)
Before — type-checked, never range-checked
# LocalPreferences.toml:  suggestion_min_score = -1.0  # loads fine
parse_equation("ZZZQX >= 0.1") … @warn "…not found in nx (10 assets) (did you mean `AAPL`?)"
# threshold gone: EVERY unknown probe now names its nearest real asset — repeated probes enumerate the universe the builders deliberately never print
After — range enforced where the type is built
StringDistanceConfig(dist, min_score) requires min_score >= 0
# ArgumentError at set_string_distance! / package load — same throw-at-load posture as the EQUATION_LIMITS preferences
# (> 1 stays legal: it is the documented "disable suggestions" sentinel)

Problem. The preference path and set_string_distance! validate min_score only as a non-Bool Real, so a zero or negative value silently removes the threshold that keeps did_you_mean from echoing a real asset name for arbitrary probes — undoing the ADR 0026 "never interpolate the universe" discipline one name per warning; latent (requires a bad project preference or setter call), not reachable with defaults.

Direction. Enforce min_score >= 0 in the StringDistanceConfig inner constructor, so every route in — setter, scoped override, load-time preference — fails closed at construction, matching the EquationLimits pattern.

  • fails closed at config load
  • one constructor guards three routes
  • preserves >1 disable sentinel
  • keeps ADR 0026 leak posture intact
ADR 0026: the suggestion threshold is what makes the fuzzy hint safe to append to the info-leak-safe builders. This finding closes the one configuration that defeats it.

Top recommendation — close the non-finite returns boundary (finding 1)

It is the review's one silent-wrong-answer path — untrusted data with a NaN or an Inf return crosses the central ReturnsResult boundary unchecked and comes out the other side as plausible, finite, wrong portfolio weights — it is reachable from the exported API with defaults, it has been open since the 2026-07-07 pass, and the fix is about four lines matching the validation the same constructor already applies to iv/ivpa.
