Ergonomics review — PortfolioOptimisers.jl

2026-07-07 · branch dev

This pass walked the public surface as a caller would: exports, the happy path from prices to weights, constructor keyword signatures, error text, and what the REPL prints back. The library's composition model is coherent and the field_dict discipline means every field is documented in ?-help — the friction concentrates in abbreviated keyword names, error messages that name internal composition paths instead of the caller's keyword, and struct printing that buries the two fields a caller set under eighty lines of defaults. Eight candidates below; items already decided in ADRs 0014/0015/0017/0023/0025–0027 are excluded.

3 · var_views means Value-at-Risk — and var_alpha's doc says the opposite

Strong naming
src/01_Base.jl:337–345 · src/13_Prior/10_EntropyPoolingPrior.jl:449–488
Before — field_dict, verbatim
:var_views   → "Value-at-risk views estimator or result."
:sigma_views → "Variance views estimator or result."
:var_alpha   → "Quantile level for variance views."
← contradicts var_views: variance views have no quantile level
After
:var_alpha → "Quantile level for value-at-risk views."
plus a one-line disambiguation on both view fields:
"var_views = Value-at-Risk (quantile), not variance —
 variance views are sigma_views."

Problem. A statistician reads var as variance and sigma as standard deviation; here both are swapped from that intuition, and the one docstring that could disambiguate (var_alpha) is itself wrong — a caller who puts a variance view in var_views gets a quantile constraint on the wrong statistic.

Direction. Fix the var_alpha entry now; add the cross-reference line to both view fields; separately weigh a value_at_risk_views keyword alias — the field names' blast radius (every EP example and user script) argues for alias-first if renamed at all.

  • doc contradiction: one-line fix
  • signal: cross-reference kills the trap
  • alias sidesteps rename blast radius
  • operator rules surfaced in docs too
Also discoverable only via ParseError: var_views accepts only ==/>= and cvar_views only == — worth stating in the field docs rather than at parse time.

Problem. A call site like JuMPOptimiser(; slv, wb, sbgt, tn, ple) fails the signal test — every read of user code routes through ?-help, and autocomplete on 38 two-to-four-letter names offers no discoverability foothold (typing thr for threshold finds nothing; the field is lt/st).

Direction. Accept long-form spellings in the keyword constructors (a name-map at construction, zero change to fields, factories, or views), with the long form listed alongside the short one in field docs — additive, so blast radius is zero and existing code is untouched.

  • signal: call sites readable unaided
  • discoverability: autocomplete finds concepts
  • blast radius: zero, additive alias
  • field_dict already holds the meanings
Scope check. The same map covers MeanRisk's r/wi/fb and ReturnsResult's nx/nf/nb/iv/ivpa — one mechanism, decided once, applied per struct. Two-name surfaces also carry a cost (which form do docs teach?); the grilling should settle a single teaching form first.

5 · Solve-failure warnings: no sub-problem name, and "please ignore" as an instruction

Worth exploring errors
src/20_Optimisation/08_Base_JuMPOptimisation.jl:736 · 13_NearOptimalCentering.jl:957 · 01_Base_Optimisation.jl:1023
Before — verbatim, as a fallback run prints them
┌ Warning: Failed to solve optimisation problem. Check `retcode.res` for details.
┌ Warning: Using fallback method. Please ignore previous optimisation failure warnings.
After — names the problem, the solver chain, and what happened next
┌ Warning: MeanRisk solve failed: all 2 solvers rejected the problem
│ (clarabel1: INFEASIBLE, highs1: TIME_LIMIT). Falling back to `fb`.
│ Inspect `res.retcode.res` for per-solver diagnostics.

Problem. The failure warn is identical from every optimiser and sub-problem, so in a NearOptimalCentering or meta-optimiser run the caller can't tell which solve failed; when a fallback rescues the run, the library asks the caller to retroactively ignore warnings instead of not emitting a misleading sequence.

Direction. Include the optimiser/sub-problem name and per-solver termination summary in the failure warn, and make the fallback path emit one self-contained warning rather than a failure + "please ignore" pair.

  • signal: which solve, which solver
  • fallback logs one coherent story
  • blast radius: warn strings only
ADR 0023. The structure of failure diagnostics (named-tuple sub-problem retcodes) is decided and done — this card is only about the human-readable warn text layered on top.

6 · The first optimisation pays a three-estimator composition cost

Worth exploring defaults
examples/1_foundations/01_Getting_Started.jl:73–96 · src/20_Optimisation/10_JuMPOptimiser.jl:330
flowchart LR
  A[caller] --> B["Solver(; name, solver, settings, check_sol)"]
  B --> C["JuMPOptimiser(; slv = …)"]
  C --> D["MeanRisk(; opt = …)"]
  D --> E["optimise(mr, rd)"]
  A2[caller] --"alias"--> D2["MeanRisk(; slv = Solver(Clarabel.Optimizer))"]
  D2 --> E2["optimise(mr, rd)"]
  classDef alias stroke:#059669,stroke-width:2px;
  class A2,D2,E2 alias

Problem. The minimum-variance base case — the field's hello-world — requires the caller to learn Solver, JuMPOptimiser, and MeanRisk and nest them correctly before the first optimise runs; slv is required with no default (correctly — the library can't pick a solver dependency), but the two wrapper layers are pure boilerplate at this point in the caller's journey.

Direction. Let the solver flow to the optimiser without the caller spelling the intermediate layer — e.g. MeanRisk(; slv = …) building the default JuMPOptimiser internally, and/or a positional Solver(Clarabel.Optimizer) for the name/settings-free case. The 90th-percentile first call sets exactly one thing: which solver.

  • friction: 3 concepts before hello-world
  • default covers the one required choice
  • blast radius: zero, additive path
  • composition model untouched underneath

Problem. Everything is exported and almost nothing is public-only (7 names), so using PortfolioOptimisers floods the namespace and autocomplete with ~609 names — the acronym layer doubles the surface a caller must scan, and the two-letter forms collide with names callers plausibly own.

Direction. Keep the domain-standard acronyms (CVaR, HRP) but move the block — or at least the ≤3-letter subset — behind an opt-in submodule or public marking, with a deprecation cycle since removal is breaking for existing scripts.

  • discoverability: autocomplete scans less noise
  • collision risk removed from caller scope
  • blast radius: existing scripts — deprecate first
  • convention: exported ⇒ teachable name

8 · w carries six documented meanings across the surface

Speculative naming
src/01_Base.jl:4–5,427–428,636,664 (field_dict entries, verbatim)
:pw → "`w`: Portfolio weights vector."
:ow → "`w`: Observation weights vector."
:w_ref → "`w`: Reference portfolio weights vector."
:w_bm_ret → "`w`: Benchmark portfolio returns vector."
:owa_w → "`w`: Optional OWA weight vector."
:ep_w → "`w`: Prior observation probability weights…"

Problem. SimpleExpectedReturns(; w=…) takes observation weights, EquilibriumExpectedReturns(; w=…) equilibrium portfolio weights, WeightsTracking(; w=…) a reference weight vector, and a tracking measure's w is benchmark returns — the internal doc keys (:pw/:ow/:w_ref/:w_bm_ret) prove the concepts are distinct, but the caller-facing name flattens them, so reading w = x at a call site tells the caller nothing.

Direction. If the long-form alias mechanism of card 4 lands, give each semantics its long name there (obs_weights, benchmark_returns, …) and leave w as the short form — a field rename is high blast radius (the res.w result convention is load-bearing and should not move).

  • signal: call sites self-describing
  • rides card 4's mechanism
  • blast radius: zero if alias-only
  • result-side res.w untouched

Top recommendation — compact Base.show (card 1)

Start with card 1. It is display-only — zero blast radius on the API, no ADR to re-litigate — and it pays out everywhere at once: every REPL inspection, every jldoctest, and every docs example currently teaches the full default plumbing tree before the caller's own choices; collapsing defaults inverts that in one place. It also softens cards 4 and 8 for free, because the abbreviated field names stop arriving eighty at a time. The error-text cards (2, 3) are the quick follow-ups: verbatim string edits with an in-repo model to copy.
