1 · Deepen the prior→measure moment binding

The same shallow family ADR 0011 just eliminated on the result side — still hand-written on the moment side.

46 × factory(r, pr::AbstractPriorResult) across 13 files · 162 × nothing_scalar_array_selector · src/19_RiskMeasures/02_Variance.jl (17), 06_XatRisk.jl (12), 19_PowerNormXatRisk.jl (14) … · src/13_Prior/*

Before — shallow, 46 copies

factory(r::Variance, pr)
  sigma = sel(r.sigma, pr.sigma)
  chol = sel(r.chol, pr.chol)
  Variance(...sigma,chol...)
factory(r::ConditionalValueatRisk, pr)
  mu = sel(r.mu, pr.mu)
  sigma = sel(r.sigma, pr.sigma)
  … rebuild
factory(r::PowerNormValueatRisk, pr)
  mu = sel(r.mu, pr.mu) … (14×)

The field→prior map is re-typed in every body. A forgotten line is a silent wrong moment — no error.

After — deep, one surface

@prior_bound sigma chol ＃ at the struct
@prior_bound mu sigma chol
factory(r, pr) ＃ generated once
for f in bound_fields(r): sel(getfield(r,f), getfield(pr,f))

One generic body reads a declared field-set. Adding a moment field = one tag, not 46 edits.

Problem. Each risk measure and prior re-lists, in a hand-written factory(r, pr) body, which moment fields it pulls from the prior — the interface (the field→prior mapping) is invisible at the struct and re-encoded 46 times. @fprop does not cover this: it propagates one value into tagged children; this selectively maps named prior-result moments onto measure fields.

Solution. A declarative binding (a @prior_bound tag, sibling to @fprop/@vprop, or a prior_bound_fields trait) that lists the bound fields at the struct; one generated factory(r, pr) runs the selector over them.

  • locality: the field→prior map sits at the struct
  • leverage: one body serves 46 call sites
  • kills the silent-wrong-moment bug class
  • interface shrinks; implementation absorbs the copies
  • test surface: one binding, not 46 bodies
  • mirrors the ADR 0011 result-side win
Touches ADR 0001/0002. Extends the propagation-macro family rather than contradicting it — the same logic that justified @fprop. Worth confirming the divergence between value-propagation and prior-binding is real enough to warrant a third tag vs. a trait.
Strong in-process · macro

2 · One delegation seam for property forwarding

A sibling gap to @fprop/@vprop: the property-forwarding decision has no macro and is hand-copied.

15 files define Base.getproperty · ~10–12 repeat the delegate-to-child template · 13_Prior/01_Base_Prior.jl, 04_HighOrderPrior.jl, 06/07/08/09/10_*Prior.jl · 20_Optimisation/08_Base_JuMPOptimisation.jl, 12_FactorRiskContribution.jl, 14_RiskBudgeting.jl

Before — same template, 10+ times

function getproperty(o::HighOrderPrior, s)
  s in propertynames(o)      ? getfield(o,s) :
  s in propertynames(o.pr)   ? getproperty(o.pr,s) :
  getfield(o,s)
end
# FactorRiskContributionResult → rr, then jr
# RiskJuMPOptimisationResult   → jr
# BayesianBlackLitterman…      → pr
# …8 more, each hand-written

Each repeats "own fields → forward to named child". No combinator. A new wrapper that forgets the template silently drops forwarding.

After — declared once

@delegate_properties HighOrderPrior => :pr
@delegate_properties FRCResult => (:rr, :jr)
@delegate_properties RiskJuMP… => :jr
one generated getproperty
＋ matching propertynames

The forwarding target is declared at the struct, like @fprop. Computed properties (:w from sol, LinearConstraint's :A_ineq) stay hand-written — they are genuine depth, not boilerplate.

Problem. Property forwarding has no macro, so ~10–12 wrappers hand-copy the same "check own fields, else delegate to a named child" body. The forwarding decision lives in the method body, invisible at the struct — and propertynames must be kept in sync by hand.

Solution. A @delegate_properties T => child macro emitting the getproperty + propertynames pair. Leave the two computed-property bodies (JuMPOptimisationResult.w, LinearConstraint) alone — they are deep.

  • locality: forwarding declared at the struct
  • leverage: one macro, 10+ wrappers
  • propertynames stays consistent for free
  • lowest-risk of the strong candidates

3 · Factor the risk-constraint-builder scaffolding

The math is genuinely distinct per measure; the wrapper around it is not.

src/20_Optimisation/20_RiskMeasureConstraints/ · 20 files, ~5,700 LOC · each set_risk_constraints! ≈ 40% identical scaffolding

Before — 20 builders, each wraps its math

cache check (haskey)
sc = get_constraint_scale(model)
‹ bespoke risk math — 5–40 lines ›
set_risk_bounds_and_expression!(…)

Cache + scale + finalise repeat in all 20. ~40% of each file is the wrapper, not the formulation.

After — one harness, 20 kernels

with_risk_constraint(model, r, i) do
  ‹ bespoke math only ›
end

Harness owns cache, scale, bounds/expression. Each file declares only the formulation that makes it distinct.

Problem. Each of 20 builders is wrapped in the same cache-check / scale-fetch / bounds-and-expression-finalise scaffolding. The wrapper is a pass-through repeated 20 times; the deletion test on the math fails (it concentrates nothing — each formulation is real), but the scaffolding is pure repetition.

Solution. A with_risk_constraint harness (higher-order builder) owning the shared steps; each file passes a kernel computing only its risk expression.

  • locality: caching/finalising fixed in one place
  • interface shrinks to the formulation kernel
  • each file reads as just its math
  • leverage: cache bug fixed once, fixed 20×
Adjacent to ADR 0004/0005. Those own the model-state seam (Category-A keys, snapshot/restore). This is the orthogonal builder-shape repetition — confirm the harness composes with the accessor interface rather than re-opening it.

4 · Name the fields of ProcessedJuMPOptimiserAttributes

src/20_Optimisation/10_JuMPOptimiser.jl · 18-field positional struct · test/test_03b_jump_model_assembly.jl re-types the 18-arg order by hand

Before — wide positional interface

interface
impl

18 positional fields (pr,wb,lt,st,lcsr,…,ret). Callers and tests must know the order. Shallow: interface ≈ implementation.

After — keyword / builder construction

interface
impl

Named fields: order stops being part of the interface. The construction site stops being a 18-slot positional contract.

Problem. An 18-field positional bundle threads through assembly; every construction and every test re-types the slot order. Order is interface; a transposed pair is a silent wrong value.

Solution. Keyword construction (@kwdef-style) or a small builder, so position drops out of the interface.

  • interface shrinks; order stops mattering
  • tests name fields, not slots

Problem. Several CV / meta result types concatenate inner results and re-expose them via broadcasting getproperty, adding no invariant. OptimisationCrossValidation is a 2-field container with no behaviour.

Solution. Run the deletion test on each: where complexity merely moves to a named tuple, collapse it; where an aggregation invariant should exist (e.g. validating inner retcodes), deepen the container to enforce it instead of leaving it to callers.

  • delete pass-throughs; concentrate the rest
  • aggregation invariant becomes a test surface
Speculative: some of these wrappers may carry dispatch value (scoring, meta-optimiser routing) that the deletion test will vindicate. Verify per type before collapsing.

Top recommendation

Start with Candidate 1 — deepen the prior→measure moment binding

It is the largest blast radius (46 hand-written factory(r, pr) bodies, 162 selector calls) and the clearest testability win: the field→prior map becomes one declared surface instead of 46 places a forgotten line silently feeds the wrong moment into an optimiser. It is also the natural continuation of ADR 0011 — the project already decided this exact "shallow copy, silent wrong slice" family is worth deepening, on the result side; this is its twin on the moment side. Candidate 2 is the lowest-risk warm-up — mechanical, behaviour-preserving, and it completes the same macro family (@fprop/@vprop → property forwarding) in one stroke.
